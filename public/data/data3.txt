Pregunta 1: **Omitido¿Cuál servicio de AWS se puede utilizar para enviar mensajes de texto (SMS) promocionales a más de 200 países de todo el mundo?**
• **Servicio de almacenamiento simple de Amazon (Amazon S3)**
• **Amazon Simple Notification Service (Amazon SNS)(Correcto)**
• **Amazon Simple Email Service (Amazon SES)**
• **Amazon Simple Queue Service (Amazon SQS)
Explicación**
**Amazon Simple Notification Service (Amazon SNS)** es un servicio de mensajería totalmente administrado para la comunicación de aplicación a aplicación (A2A) y de aplicación a persona (A2P). La funcionalidad A2P le permite enviar mensajes a los usuarios a escala a través de SMS, push móvil y correo electrónico.
Amazon SNS le permite enviar mensajes o notificaciones directamente a los usuarios con mensajes de texto SMS a más de 200 países. Además, puede marcar sus mensajes SMS como transaccionales para optimizar la entrega confiable, o puede marcarlos como promocionales para optimizar el ahorro de costos. Los mensajes SMS que llevan mensajes de marketing deben estar marcados como promocionales. Amazon SNS garantiza que los mensajes promocionales se envíen a través de rutas que tienen una fiabilidad de entrega razonable pero que son sustancialmente más baratas que las rutas más fiables.

***Las otras opciones son incorrectas:***

***"Amazon Simple Email Service (Amazon SES)" es incorrecto.*** Amazon SES solo se puede utilizar para enviar **correos electrónicos**, no mensajes de texto (SMS). Amazon SES es un servicio de envío de correo electrónico basado en la nube diseñado para ayudar a los especialistas en marketing digital y desarrolladores de aplicaciones a enviar correos electrónicos de marketing, notificaciones y transacciones.

***"Amazon Simple Queue Service (Amazon SQS)" es incorrecto.*** Amazon SQS es un servicio de cola de mensajes altamente confiable que permite la comunicación asincrónica basada en mensajes entre componentes distribuidos de una aplicación. Con SQS, puede enviar, almacenar y recibir mensajes entre componentes de software en cualquier volumen, sin perder mensajes ni requerir que otros servicios estén disponibles.

***"Amazon Simple Storage Service" es incorrecto.*** Amazon Simple Storage Service (Amazon S3) es un servicio de almacenamiento de objetos.

**Referencias:**
https://aws.amazon.com/sns/

Pregunta 2: **Omitido¿Cuáles de los siguientes son casos de uso para Amazon S3? (Elija DOS)**
• [ ] **Alojamiento de sitios web estáticos(Correcto)**
• [ ] **Un almacén multimedia para el servicio CloudFront(Correcto)**
• [ ] **Alojamiento de sitios web que requieren una alta utilización sostenida de la CPU**
• [ ] **Procesamiento de flujos de datos a cualquier escala**
• [ ] **Almacenamiento rentable de bases de datos y registros
Explicación**
Puede alojar un sitio web estático en Amazon Simple Storage Service (Amazon S3). En un sitio web estático, las páginas web individuales incluyen contenido estático. También pueden contener scripts del lado del cliente. Para alojar un sitio web estático, configure un bucket de Amazon S3 para el alojamiento de sitios web, permita el acceso de lectura pública y, a continuación, cargue el contenido del sitio web en el bucket. Por el contrario, un sitio web dinámico se basa en el procesamiento del lado del servidor, incluidos los scripts del lado del servidor como PHP, JSP o ASP.NET. Amazon S3 no admite secuencias de comandos del lado del servidor. Amazon Web Services (AWS) también tiene recursos para alojar sitios web dinámicos como Amazon EC2.
Amazon S3 es una excelente instalación de almacenamiento para sus activos multimedia. Es infinitamente escalable, tiene redundancia incorporada y está disponible para usted en una base de pago por uso. Por ejemplo, si desea entregar o transmitir archivos de vídeo a sus usuarios globales, todo lo que necesita hacer es colocar su contenido en un bucket de S3 y crear una distribución de CloudFront que apunte al bucket. El reproductor de vídeo del usuario utilizará las URL de CloudFront para solicitar el archivo de vídeo. La solicitud se dirigirá a la mejor ubicación de borde, según la ubicación del usuario. La red de entrega de contenido (CDN) de Amazon Cloudfront servirá el vídeo desde su caché, obteniéndolo del bucket de S3 si aún no se ha almacenado en caché. La CDN almacena en caché el contenido en las ubicaciones perimetrales para una entrega de vídeo coherente, de baja latencia y alto rendimiento.

***Las otras opciones son incorrectas:***

***"Almacenamiento rentable de bases de datos y registros" es incorrecto.*** Amazon S3 se puede utilizar para almacenar archivos de registro, imágenes, vídeos (o cualquier contenido estático), pero no bases de datos. Las bases de datos y los sitios web dinámicos requieren almacenamiento de información a nivel de bloque (como EBS). S3 es un almacenamiento a nivel de objeto, no un almacenamiento a nivel de bloque. El almacenamiento a nivel de objeto tiene E/S limitadas y, por lo tanto, no es adecuado para su uso como almacén de bases de datos.

***"Alojar sitios web que requieren una alta utilización sostenida de la CPU" es incorrecto.*** S3 solo se puede utilizar para alojar sitios web estáticos.

***"Procesar flujos de datos a cualquier escala" es incorrecto.*** S3 no es un servicio informático.

**Referencias:**
https://docs.aws.amazon.com/AmazonS3/latest/dev/WebsiteHosting.html
https://aws.amazon.com/cloudfront/streaming/

Pregunta 4: **Omitido¿Cuál de las siguientes opciones NO es una característica de Amazon Elastic Compute Cloud (Amazon EC2)?**
• **Amazon EC2 elimina la necesidad de invertir en hardware por adelantado**
• **Amazon EC2 se considera un servicio web sin servidor(Correcto)**
• **Amazon EC2 ofrece informática escalable**
• **Amazon EC2 puede lanzar tantos o tan pocos servidores virtuales como sea necesario
Explicación**
***"Amazon EC2 se considera un servicio web sin servidor"*** no es una característica de Amazon EC2 y, por lo tanto, es la opción correcta. Serverless permite a los clientes transferir más responsabilidades operativas a AWS. Serverless permite a los clientes crear y ejecutar aplicaciones y servicios sin pensar en los servidores. Serverless elimina las tareas de administración de infraestructura, como el aprovisionamiento de servidores o clústeres, la aplicación de parches, el mantenimiento del sistema operativo y el aprovisionamiento de capacidad.
Amazon EC2 no es un servicio sin servidor. Las instancias EC2 son servidores virtuales en la nube. Amazon Elastic Compute Cloud (Amazon EC2) proporciona capacidad informática escalable en la nube de Amazon Web Services (AWS). El uso de Amazon EC2 elimina la necesidad de invertir en hardware por adelantado, para que pueda desarrollar e implementar aplicaciones más rápido. Puede utilizar Amazon EC2 para lanzar tantos o tan pocos servidores virtuales como necesite, configurar la seguridad y las redes, y administrar el almacenamiento. Amazon EC2 le permite escalar hacia arriba o hacia abajo para gestionar los cambios en los requisitos o los picos de popularidad, lo que reduce la necesidad de pronosticar el tráfico.

**Referencias:**
http://docs.aws.amazon.com/AWSEC2/latest/UserGuide/concepts.html

Pregunta 5: **Omitido¿Por qué cada región de AWS contiene varias zonas de disponibilidad?**
• **Múltiples zonas de disponibilidad da como resultado un menor costo total en comparación con la implementación en una sola zona de disponibilidad**
• **Múltiples zonas de disponibilidad permiten la replicación de datos y el alcance global**
• **Varias zonas de disponibilidad dentro de una región aumentan la capacidad de almacenamiento disponible en esa región**
• **Múltiples zonas de disponibilidad le permiten crear arquitecturas resistentes y de alta disponibilidad(Correcto)
Explicación**
La **resiliencia** es la capacidad de una arquitectura para continuar brindando la misma calidad de servicio, incluso si algunos de sus recursos se vuelven inaccesibles. La implementación de sus recursos en varias zonas de disponibilidad le ofrece la capacidad de operar aplicaciones de producción y bases de datos que son más **resistentes, de alta disponibilidad y escalables** de lo que sería posible desde un solo centro de datos.

***Las otras opciones son incorrectas:***

***"Múltiples zonas de disponibilidad dentro de una región aumenta la capacidad de almacenamiento disponible en esa región" es incorrecto.*** En AWS, tiene una capacidad de almacenamiento prácticamente ilimitada independientemente de las regiones o zonas de disponibilidad de una región.

***"Múltiples zonas de disponibilidad resulta en un menor costo total en comparación con la implementación en una sola zona de disponibilidad" es incorrecto.*** La implementación de sus recursos en varias zonas de disponibilidad no tiene beneficios de costos.

***"Múltiples zonas de disponibilidad permite la replicación de datos y el alcance global" es incorrecto.*** Múltiples zonas de disponibilidad dentro de una región permiten la replicación de datos, pero no el alcance global.

Las instancias de spot son la opción más rentable si es flexible en cuanto a cuándo se ejecutan sus aplicaciones y si sus aplicaciones se pueden interrumpir. ***"Instancias reservadas" es incorrecto.*** Las instancias reservadas proporcionan un descuento (hasta un 75 %) en comparación con la instancia bajo demanda. Incluso si la pregunta indica que la empresa necesita las instancias durante un año, la mejor respuesta debería seguir siendo las instancias puntuales, ya que ofrecen una mayor reducción de costos generales (hasta un 90 %) que las instancias reservadas.

***"Instancias dedicadas" es incorrecto.*** Las instancias dedicadas se utilizan cuando desea que las instancias se aíslen físicamente en el nivel de hardware del host de las instancias que pertenecen a otras cuentas de AWS.

**AWS Service Catalog** permite a las organizaciones crear y administrar catálogos de servicios de TI aprobados para su uso en AWS. Estos servicios de TI pueden incluir todo, desde imágenes de máquinas virtuales, servidores, software y bases de datos hasta arquitecturas de aplicaciones completas de varios niveles. AWS Service Catalog le permite administrar de forma centralizada los servicios de TI implementados comúnmente y le ayuda a lograr una gobernanza coherente y cumplir con sus requisitos de conformidad, al tiempo que permite a los usuarios implementar rápidamente solo los servicios de TI aprobados que necesitan.

***"Permite a los desarrolladores implementar infraestructura en AWS utilizando lenguajes de programación conocidos" es incorrecto.*** AWS Cloud Development Kit (AWS CDK) es el servicio que permite a los desarrolladores modelar e implementar infraestructura en AWS utilizando lenguajes de programación conocidos. AWS Cloud Development Kit (AWS CDK) es un marco de desarrollo de software de código abierto para **definir la infraestructura de la nube como código con lenguajes de programación modernos e implementarlo a través de AWS CloudFormation**. AW CDK le permite utilizar sus habilidades y herramientas de programación existentes, y aplicarlas a la tarea de construir infraestructura en la nube. AWS CDK está disponible con carácter general en JavaScript, TypeScript, Python, Java y C#.
Información adicional:
¿Cuál es la relación entre AWS CDK y AWS CloudFormation?
Puede pensar en AWS CDK como un kit de herramientas centrado en el desarrollador que aprovecha toda la potencia de los lenguajes de programación modernos para definir su infraestructura de AWS como código. El CDK en realidad se basa en AWS CloudFormation y lo utiliza como motor para aprovisionar recursos de AWS. En lugar de usar un lenguaje declarativo como JSON o YAML para definir su infraestructura (como es el caso de CloudFormation), el CDK le permite hacerlo en su lenguaje de programación imperativo favorito. Esto incluye lenguajes como JavaScript, TypeScript, Java, C# y Python. Cuando se ejecutan aplicaciones de AWS CDK, se compilan en plantillas JSON/YAML de CloudFormation completamente formadas que luego se envían al servicio CloudFormation para su aprovisionamiento.

Pregunta 11: **Omitido¿Cuál de las siguientes opciones se utiliza para controlar el tráfico de red en AWS? (Elija DOS)**
• [ ] **Security Groups (Grupos de seguridad)(Correcto)**
• [ ] **Access Keys (Llaves de acceso)**
• [ ] **IAM Policies (Políticas de IAM)**
• [ ] **Key Pairs (Pares de claves)**
• [ ] **Network Access Control Lists (NACLs) (Listas de control de acceso a la red)(Correcto)
Explicación**
Puede controlar el tráfico de red en AWS configurando grupos de seguridad, listas de control de acceso a la red y tablas de enrutamiento.
1- **Grupos de seguridad**: Actúa como firewall para las instancias Amazon EC2 asociadas, controlando tanto el tráfico entrante como el saliente a nivel de instancia.
2- **Listas de control de acceso a la red (ACL)**: Actúan como un firewall para las subredes asociadas, controlando tanto el tráfico entrante como el saliente a nivel de subred.
3- **Tablas de enrutamiento:** Una tabla de enrutamiento contiene un conjunto de reglas, llamadas rutas, que se utilizan para determinar hacia dónde se dirige el tráfico de red.

**Nota:**
**El control del tráfico de red utilizando cualquiera de los métodos anteriores es responsabilidad del cliente.**

***Las otras opciones son incorrectas:***

***"Claves de acceso" es incorrecto.*** Las claves de acceso son credenciales a largo plazo para un usuario de IAM o el usuario raíz de la cuenta de AWS. Las claves de acceso le permiten interactuar con los servicios de AWS mediante programación mediante la AWS CLI o el SDK de AWS.

***"Políticas de IAM" es incorrecto.*** De forma predeterminada, los usuarios de IAM no tienen permiso para crear o modificar recursos en AWS. Las políticas de IAM se utilizan para conceder a los usuarios de IAM permiso para usar los recursos específicos y las acciones de API que necesitarán.

***"Pares de claves" es incorrecto.*** Amazon EC2 utiliza criptografía de clave pública para cifrar y descifrar la información de inicio de sesión. La criptografía de clave pública utiliza una clave pública para cifrar un dato y, a continuación, el destinatario utiliza la clave privada para descifrar los datos. Las claves públicas y privadas se conocen como un par de claves. La criptografía de clave pública le permite acceder de forma segura a sus instancias utilizando una clave privada en lugar de una contraseña.

Pregunta 13: **Omitido¿Cuál de las siguientes declaraciones describe la agilidad de la nube de AWS?**
• **AWS le permite pagar por adelantado para reducir los costos**
• **AWS le permite aprovisionar recursos en minutos(Correcto)**
• **AWS proporciona hardware personalizable al menor costo posible**
• **AWS le permite alojar sus aplicaciones en varias regiones de todo el mundo
Explicación**
En un entorno de computación en la nube, los nuevos recursos de TI están a solo un clic de distancia, lo que significa que reduce el tiempo para poner esos recursos a disposición de sus desarrolladores de semanas (o meses en algunos casos) a solo minutos. Las políticas de control de servicios de AWS (o políticas de AWS Organizations) son un tipo de política de organización que puede utilizar para administrar los permisos de todas las cuentas de su organización. Los SCP ofrecen un control central sobre el máximo de permisos disponibles para todas las cuentas de miembros de su organización. Los SCP le ayudan a garantizar que las cuentas de los miembros se mantengan dentro de las directrices de control de acceso de su organización. Información adicional:
**¿Cuál es la diferencia entre una política de control de servicios (SCP) de AWS Organizations y una política de IAM?**
Una política de IAM proporciona un control granular sobre lo que pueden hacer los usuarios y roles de cuentas individuales. Los permisos resultantes son la intersección lógica de lo permitido por AWS Organizations en el nivel de cuenta y los permisos que IAM concede explícitamente en el nivel de usuario o rol dentro de esa cuenta. En otras palabras, el usuario solo puede acceder a lo permitido tanto por las políticas de AWS Organizations como por las políticas de IAM. Si alguno de los dos bloquea una operación, el usuario no puede acceder a esa operación. Por ejemplo, si un SCP aplicado a una cuenta indica que las únicas acciones permitidas son las acciones de Amazon EC2 y los permisos de una entidad de seguridad (usuario o rol de IAM) en la misma cuenta de AWS permiten tanto acciones de EC2 como acciones de Amazon S3, la entidad de seguridad solo puede obtener acceso a las acciones de EC2.

Hay **cinco principios de diseño** para la eficiencia del rendimiento en la nube:
1- **Democratizar las tecnologías avanzadas**: Las tecnologías que son difíciles de implementar pueden ser más fáciles de consumir al llevar ese conocimiento y complejidad al dominio del proveedor de la nube. En lugar de que su equipo de TI aprenda a alojar y ejecutar una nueva tecnología, simplemente pueden consumirla como un servicio. Por ejemplo, las bases de datos NoSQL, la transcodificación de medios y el aprendizaje automático son tecnologías que requieren experiencia que no está dispersa de manera uniforme en toda la comunidad técnica. En la nube, estas tecnologías se convierten en servicios que su equipo puede consumir mientras se centra en el desarrollo de productos en lugar del aprovisionamiento y la gestión de recursos.
2- **Globalícese en minutos**: implemente fácilmente su sistema en varias regiones de todo el mundo con solo unos pocos clics. Esto le permite proporcionar una latencia más baja y una mejor experiencia para sus clientes a un costo mínimo.
3- **Utilice arquitecturas sin servidor:** En la nube, las arquitecturas sin servidor eliminan la necesidad de ejecutar y mantener servidores para llevar a cabo actividades informáticas tradicionales. Por ejemplo, los servicios de almacenamiento pueden actuar como sitios web estáticos, eliminando la necesidad de servidores web, y los servicios de eventos pueden alojar su código por usted. Esto no solo elimina la carga operativa de administrar estos servidores, sino que también puede reducir los costos transaccionales porque estos servicios administrados operan a escala de nube.
4- **Experimente con más frecuencia:** Con recursos virtuales y automatizables, puede realizar rápidamente pruebas comparativas utilizando diferentes tipos de instancias, almacenamiento o configuraciones.
5- **Afinidad mecánica:** Utilice el enfoque tecnológico que mejor se alinee con lo que está tratando de lograr. Por ejemplo, tenga en cuenta los patrones de acceso a los datos al seleccionar enfoques de base de datos o almacenamiento.

Pregunta 39: **OmitidoUna empresa planea migrar una gran cantidad de datos archivados a AWS. Los datos archivados deben mantenerse durante un período de 5 años y deben ser recuperables dentro de las 5 horas posteriores a una solicitud. ¿Cuál es el servicio de almacenamiento de AWS más rentable de usar?**
• **Amazon EFS Infrequent Access (Acceso poco frecuente de Amazon EFS)**
• **Amazon EBS Infrequent Access (Acceso poco frecuente de Amazon EBS)**
• **Amazon S3 Glacier(Correcto)**
• **Amazon S3 Standard
Explicación**
Los clientes de AWS pueden utilizar **Amazon S3 Glacier** o **Amazon S3 Glacier Deep Archive** para realizar copias de seguridad de grandes cantidades de datos a costos muy bajos.
**Amazon S3 Glacier** es un servicio de almacenamiento en la nube seguro, duradero y de muy bajo costo para el archivado de datos y la copia de seguridad a largo plazo. Está diseñado para ofrecer una durabilidad del 99,999999999% y proporciona capacidades integrales de seguridad y cumplimiento que pueden ayudar a cumplir incluso con los requisitos reglamentarios más estrictos.
**Amazon S3 Glacier Deep Archive** es la clase de almacenamiento de menor costo de Amazon S3 que admite la retención a largo plazo y la preservación digital de los datos a los que se puede acceder una o dos veces al año.
Elegir entre S3 Glacier y S3 Glacier Deep Archive depende de la rapidez con la que deba recuperar los datos. Con S3 Glacier, puede recuperar sus datos en **unos pocos minutos a varias horas (1-5 minutos a 12 horas),** mientras que con S3 Glacier Deep Archive, el período mínimo de recuperación es de 12 horas.

***Las otras opciones son incorrectas:***

***"Acceso poco frecuente de Amazon EFS" es incorrecto.*** Amazon Elastic File System (Amazon EFS) no es una solución rentable para el archivado de datos. Amazon EFS es un servicio **de** almacenamiento de archivos para su uso con Amazon Compute (EC2, contenedores, sin servidor) y servidores locales. Amazon EFS proporciona una interfaz de sistema de archivos, semántica de acceso al sistema de archivos (como coherencia sólida y bloqueo de archivos) y almacenamiento accesible simultáneamente para hasta miles de instancias de Amazon EC2.
**[¿Qué es el acceso poco frecuente de Amazon EFS?](https://aws.amazon.com/efs/features/infrequent-access/)**
Amazon EFS Standard-Infrequent Access (EFS Standard-IA) y Amazon EFS One Zone-Infrequent Access (EFS One Zone-IA) son clases de almacenamiento que proporcionan un precio/rendimiento optimizado para los archivos a los que no se accede todos los días, con precios **de almacenamiento hasta un 92 % más bajos** en comparación con las clases de almacenamiento amazon EFS Standard (EFS Standard) y Amazon EFS One Zone (EFS One Zone), respectivamente. Para comenzar a utilizar las clases de almacenamiento de acceso poco frecuente (IA), simplemente habilite la administración del ciclo de vida de Amazon EFS para su sistema de archivos seleccionando una política de ciclo de vida que se adapte a sus necesidades. Amazon EFS moverá sus archivos de forma automática y transparente a la clase de almacenamiento regional EFS Standard-IA de menor costo o a la clase de almacenamiento EFS One Zone-IA en función de la última vez que se accedió a ellos. No tiene que preocuparse por cuáles de sus archivos se utilizan activamente y a cuáles se accede con poca frecuencia.

***"Acceso poco frecuente de Amazon EBS" es incorrecto.*** Amazon EBS no es una solución rentable para el archivado de datos. Amazon EBS proporciona volúmenes **de almacenamiento a nivel de bloque** para su uso con instancias Amazon EC2 y RDS. Amazon EBS no ofrece niveles de almacenamiento para datos a los que se accede con menos frecuencia. Las capas de almacenamiento de acceso poco frecuente solo están disponibles para Amazon S3 y Amazon EFS.

***"Amazon S3 Standard" es incorrecto.*** Amazon S3 Standard no es una solución rentable para el archivado de datos. Amazon S3 Standard es un almacenamiento de objetos de uso general para datos activos a los que se accede con frecuencia con **acceso de milisegundos**. Los casos de uso de S3 Standard incluyen: aplicaciones en la nube, sitios web dinámicos, distribución de contenido, aplicaciones móviles y de juegos, y análisis de big data.

Información adicional:
En S3, solo podemos alojar sitios web **estáticos o activos estáticos de un sitio web dinámico** (como imágenes, archivos de audio, archivos de video, etc.).
Un sitio web dinámico se basa en el procesamiento del lado del servidor y utiliza scripts del lado del servidor como PHP, JSP o ASP.NET. Amazon S3 no admite secuencias de comandos del lado del servidor y no se puede utilizar para alojar sitios web dinámicos. AWS cuenta con recursos informáticos para alojar sitios web dinámicos como Amazon EC2 o Lambda.

Pregunta 40: **OmitidoUn cliente planea mover miles de millones de imágenes y videos para almacenarlos en Amazon S3. El cliente tiene aproximadamente 60 Petabytes de datos para mover. ¿Cuál de los siguientes servicios de AWS es la mejor opción para transferir los datos a AWS?**

Puede transferir hasta 100 Petabytes (PB) por moto de nieve, un contenedor de envío robusto de 45 pies de largo, tirado por un camión semirremolque. Snowmobile facilita el traslado de volúmenes masivos de datos a la nube, incluidas bibliotecas de video, repositorios de imágenes o incluso una migración completa del centro de datos. Los clientes pueden transferir hasta 80 terabytes por bola de nieve. Para conjuntos de datos de menos de 10 Petabytes o distribuidos en varias ubicaciones, debe utilizar Snowball.

Amazon S3 Transfer Acceleration permite transferencias rápidas de archivos a largas distancias entre su cliente y un bucket de S3. Transfer Acceleration aprovecha las ubicaciones de borde distribuidas globalmente de Amazon CloudFront. A medida que los datos llegan a una ubicación de borde, los datos se enrutan a Amazon S3 a través de una ruta de red optimizada. Si está cargando en un bucket centralizado desde ubicaciones geográficamente dispersas, o si transfiere regularmente GB o TB de datos a través de continentes, puede ahorrar horas o días de tiempo de transferencia de datos con S3 Transfer Acceleration.

Pregunta 57: **Omitido¿Qué son los controles compartidos de AWS?**
• **Controles que un cliente hereda de AWS**
• **Controles que son responsabilidad exclusiva del cliente en función de la aplicación que está implementando dentro de los servicios de AWS**
• **Controles en los que el cliente y AWS colaboran juntos para proteger la infraestructura**
• **Controles que se aplican tanto a la capa de infraestructura como a las capas de cliente(Correcto)
Explicación**
**Los controles compartidos son controles que se aplican tanto a la capa de infraestructura como a las capas de cliente**, pero en contextos o perspectivas completamente separados. En un control compartido, AWS proporciona los requisitos para la infraestructura y el cliente debe proporcionar su propia implementación de control dentro de su uso de los servicios de AWS. Algunos ejemplos son:
** Administración de parches: AWS es responsable de aplicar parches a los hosts subyacentes y corregir fallas dentro de la infraestructura, pero los clientes son responsables de aplicar parches a sus aplicaciones y sistemas operativos invitados.
** Administración de la configuración: AWS mantiene la configuración de sus dispositivos de infraestructura, pero un cliente es responsable de configurar sus propios sistemas operativos, bases de datos y aplicaciones invitados.
** Sensibilización y capacitación: AWS capacita a los empleados de AWS, pero un cliente debe capacitar a sus propios empleados.

***Las otras opciones son incorrectas:***

***"Controles que son responsabilidad exclusiva del cliente en función de la aplicación que está implementando dentro de los servicios de AWS" es incorrecto*** porque se refiere a controles "específicos del cliente".

***"Controles que un cliente hereda de AWS" es incorrecto*** porque hace referencia a "Controles heredados".

***"Controles en los que el cliente y AWS colaboran juntos para proteger la infraestructura" es incorrecto.*** La seguridad de la infraestructura es responsabilidad de AWS, no del cliente.

Pregunta 65: **OmitidoUna empresa necesita realizar un seguimiento de los cambios en los recursos utilizando el historial de llamadas a la API. ¿Qué servicio de AWS puede ayudar a la empresa a lograr este objetivo?**
• **Amazon CloudWatch**
• **AWS CloudTrail(Correcto)**
• **AWS CloudFormation**
• **AWS Config
Explicación**
**AWS CloudTrail** es un servicio web que registra las llamadas a la API de AWS para su cuenta y le entrega archivos de registro. La información registrada incluye la identidad del llamador de la API, la hora de la llamada a la API, la dirección IP de origen del llamador de la API, los parámetros de solicitud y los elementos de respuesta devueltos por el servicio de AWS. Con CloudTrail, puede obtener un historial de las llamadas a la API de AWS para su cuenta, incluidas las llamadas a la API realizadas con la consola de administración de AWS, los SDK de AWS, las herramientas de línea de comandos y los servicios de AWS de nivel superior (como AWS CloudFormation). El historial de llamadas a la API de AWS producido por CloudTrail permite el análisis de seguridad, el seguimiento de cambios de recursos y la auditoría de conformidad.

***Las otras opciones son incorrectas:***

***"AWS Config" es incorrecto.***
Tanto AWS Config como AWS CloudTrail se pueden utilizar para realizar un seguimiento de los cambios en los recursos, y es muy importante distinguirlos. AWS Config se utiliza para supervisar y auditar los cambios en los recursos de AWS y le permite automatizar la evaluación de las configuraciones registradas de un recurso específico con respecto a las configuraciones deseadas. AWS CloudTrail registra la actividad de la API del usuario en su cuenta y le permite obtener acceso a la información sobre esta actividad. Obtiene detalles completos sobre las acciones de la API, como la identidad de la persona que llama, la hora de la llamada a la API, los parámetros de solicitud y los elementos de respuesta devueltos por el servicio de AWS.
AWS Config registra los detalles de configuración puntuales de sus recursos de AWS como elementos de configuración (CI). Puede utilizar un CI para responder "¿Cómo era mi recurso de AWS?" en un momento dado. Puede utilizar AWS CloudTrail para responder "¿Quién realizó una llamada a la API para modificar este recurso?" Por ejemplo, puede utilizar la consola de administración de AWS para AWS Config para detectar que el grupo de seguridad "Production-DB" se configuró incorrectamente en el pasado. Con la información integrada de AWS CloudTrail, puede identificar qué grupo de seguridad "Production-DB" del usuario ha configurado incorrectamente. En resumen, AWS Config proporciona información sobre los cambios realizados en un recurso y AWS CloudTrail proporciona información sobre quién realizó esos cambios.

***"AWS CloudFormation" es incorrecto.*** AWS CloudFormation es un servicio que le permite utilizar un archivo de texto simple para modelar y aprovisionar, de manera automatizada y segura, todos los recursos necesarios para sus aplicaciones en todas las regiones y cuentas.

***"Amazon CloudWatch" es incorrecto.*** Amazon CloudWatch se utiliza para supervisar y recopilar métricas personalizadas y granulares sobre sus recursos de AWS.

**Referencias:**
https://d1.awsstatic.com/whitepapers/aws-overview.pdf

Pregunta 1: **Omitido¿Cuál de los siguientes servicios de AWS puede utilizarse para prever el uso y los costes de tu cuenta de AWS?**
• **AWS Cost Explorer(Correcto)**
• **AWS Pricing Calculator**
• **AWS Cost & Usage Report (AWS CUR)**
• **AWS Budgets
Explicación**
Opción correcta:
**AWS Cost Explorer**
AWS Cost Explorer tiene una interfaz fácil de usar que te permite visualizar, comprender y administrar tus costes y uso de AWS a lo largo del tiempo. AWS Cost Explorer incluye un informe predeterminado que te ayuda a visualizar los costes y el uso asociados a tus cinco principales servicios de AWS que generan costes, y te ofrece un desglose detallado de todos los servicios en la vista de tabla. Los informes te permiten ajustar el intervalo de tiempo para ver datos históricos que se remontan hasta doce meses atrás para comprender mejor tus tendencias de costes. AWS Cost Explorer también admite previsiones para hacerte una mejor idea de cómo pueden ser tus costes y tu uso en el futuro, de modo que puedas planificar.Características AWS Cost Explorer:  vía - https://aws.amazon.com/aws-cost-management/aws-cost-explorer/
Opciones incorrectas:
**AWS Cost & Usage Report (AWS CUR)** - AWS Cost & Usage Report (AWS CUR) contiene el conjunto más completo de datos de costes y uso disponible. Puedes utilizar AWS Cost & Usage Report para publicar tus informes de facturación de AWS en un bucket de Amazon Simple Storage Service (Amazon S3) de tu propiedad. Puedes recibir informes que desglosen tus costes por hora o mes, por producto o recurso de producto, o por etiquetas que tú mismo definas. AWS actualiza el informe en tu bucket una vez al día en un formato de valores separados por comas (CSV). Los Informes de Costes y Uso de AWS no pueden prever el coste y uso de tu cuenta de AWS.
**AWS Budgets** - AWS Budgets te ofrece la posibilidad de establecer presupuestos personalizados que te avisen cuando tus costes o tu utilización superen (o se prevea que superen) la cantidad presupuestada. También puedes utilizar AWS Budgets para establecer objetivos de utilización o cobertura de reservas y recibir alertas cuando tu utilización caiga por debajo del umbral que definas. Los presupuestos pueden crearse a nivel mensual, trimestral o anual, y puedes personalizar las fechas de inicio y fin. Puedes refinar aún más tu presupuesto para realizar un seguimiento de los costes asociados a múltiples dimensiones, como el servicio de AWS, la cuenta vinculada, la etiqueta y otras. AWS Budgets no pueden prever el coste y el uso de tu cuenta de AWS.
**AWS Pricing Calculator** - AWS Pricing Calculator te permite explorar los servicios de AWS y crear una estimación del coste de tus casos de uso en AWS. Puedes modelar tus soluciones antes de construirlas, explorar los puntos de precio y los cálculos que hay detrás de tu estimación, y encontrar los tipos de instancias disponibles y las condiciones contractuales que se ajusten a tus necesidades. Esto te permite tomar decisiones informadas sobre el uso de AWS. Puedes planificar tus costes y uso de AWS o calcular el precio de configurar un nuevo conjunto de instancias y servicios. No puedes utilizar este servicio para prever el coste y el uso de tu cuenta de AWS.
Referencia:
https://aws.amazon.com/aws-cost-management/aws-cost-explorer/

Pregunta 2: **OmitidoUna empresa de medios de comunicación sube sus archivos multimedia (audio y vídeo) a un bucket centralizado de Amazon Simple Storage Service (Amazon S3) desde ubicaciones geográficamente dispersas. ¿Cuál de las siguientes soluciones puede utilizar la empresa para optimizar la velocidad de transferencia?**
• **Amazon S3 Transfer Acceleration (S3TA)(Correcto)**
• **AWS Direct Connect**
• **Amazon CloudFront**
• **AWS Global Accelerator
Explicación**
Opción correcta:
**Amazon S3 Transfer Acceleration (S3TA)**
Amazon S3 Transfer Acceleration (S3TA) permite transferencias rápidas, sencillas y seguras de archivos a través de largas distancias entre tu cliente y tu bucket de Amazon S3. La Aceleración de transferencias de S3 aprovecha las ubicaciones de borde de AWS distribuidas globalmente de Amazon CloudFront. Cuando los datos llegan a una ubicación de borde de AWS, se dirigen a tu bucket de Amazon S3 a través de una ruta de red optimizada. Amazon S3 Transfer Acceleration está diseñada para optimizar las velocidades de transferencia desde cualquier parte del mundo a los buckets de S3. Si estás subiendo a un bucket centralizado desde ubicaciones geográficamente dispersas, o si transfieres regularmente GBs o TBs de datos a través de continentes, puedes ahorrar horas o días de tiempo de transferencia de datos con la Aceleración de Transferencia de S3.Ventajas de Amazon S3 Transfer Acceleration (S3TA):  vía - https://aws.amazon.com/s3/transfer-acceleration/
Opciones incorrectas:
**Amazon CloudFront** - Amazon CloudFront es un servicio rápido de red de entrega de contenidos (CDN) que entrega de forma segura datos, vídeos, aplicaciones y API a clientes de todo el mundo con baja latencia y altas velocidades de transferencia, todo ello en un entorno fácil de usar para los desarrolladores. CloudFront se utiliza para la entrega de contenidos más que para la carga de datos. CloudFront almacena datos en caché y una solicitud posterior de una página web no irá al servidor de origen, sino que se servirá desde la caché. La Aceleración de Transferencia S3 es una mejor opción para el caso de uso dado.
**AWS Direct Connect** - AWS Direct Connect es una solución de servicio en el Cloud que facilita el establecimiento de una conexión de red dedicada desde tus instalaciones a AWS. Puedes utilizar AWS Direct Connect para establecer una interfaz virtual privada desde tu red local directamente a tu Amazon VPC. Esta conexión privada tarda al menos un mes en completarse. No puedes utilizar Direct Connect para optimizar las cargas de medios en S3.
**AWS Global Accelerator** - AWS Global Accelerator es un servicio que mejora la disponibilidad y el rendimiento de tus aplicaciones con usuarios locales o globales. Proporciona direcciones IP estáticas que actúan como punto de entrada fijo a los puntos finales de tus aplicaciones en una o varias regiones de AWS, como tus Application Load Balancer, Network Load Balancers o instancias de Amazon EC2. De forma similar a CloudFront, utiliza la red global de AWS y las ubicaciones de borde para mejorar el rendimiento. Es un potenciador del rendimiento global más que un acelerador de la velocidad de carga. No puedes utilizar el Acelerador Global para optimizar las subidas de medios a S3.
Referencia:
https://aws.amazon.com/s3/transfer-acceleration/

**Datos de la AMI de la instancia de Amazon EC2**

Pregunta 4: **Omitido¿Cuál de las siguientes opciones es la más adecuada para equilibrar la carga del tráfico HTTP y HTTPS?**
• **Network Load Balancer**
• **AWS Auto Scaling**
• **Application Load Balancer(Correcto)**
• **System Load Balancer
Explicación**
Opción correcta:
**Application Load Balancer**
Elastic Load Balancer (ELB) distribuye automáticamente el tráfico entrante de la aplicación entre múltiples destinos, como instancias de Amazon EC2, contenedores, direcciones IP y funciones Lambda. Puede gestionar la carga variable del tráfico de tu aplicación en una única zona de disponibilidad o en varias zonas de disponibilidad. Elastic Load Balancer (ELB) ofrece tres tipos de Load Balancers, todos ellos con la alta disponibilidad, el escalado automático y la sólida seguridad necesarios para que tus aplicaciones sean tolerantes a fallos. vía - https://aws.amazon.com/elasticloadbalancing/
El Application Load Balancer se utiliza para equilibrar la carga del tráfico HTTP y HTTPS y proporciona un enrutamiento avanzado de peticiones orientado a la entrega de arquitecturas de aplicaciones modernas, incluidos microservicios y contenedores.
Opciones incorrectas:
**Network Load Balancer** - Network Load Balancer es el más adecuado para equilibrar la carga de tráfico de Protocolo de Control de Transmisión (TCP), Protocolo de Datagramas de Usuario (UDP) y Seguridad de la Capa de Transporte (TLS) cuando se requiere un rendimiento extremo.
**AWS Auto Scaling** - AWS Auto Scaling monitoriza tus aplicaciones y ajusta automáticamente la capacidad para mantener un rendimiento constante y predecible al menor coste posible. Con AWS Auto Scaling, es fácil configurar el escalado de aplicaciones para múltiples recursos a través de múltiples servicios en cuestión de minutos. El servicio proporciona una interfaz de usuario sencilla y potente que te permite crear planes de escalado para recursos como instancias de Amazon EC2 y Spot Fleets, tareas de Amazon ECS, tablas e índices de Amazon DynamoDB y réplicas de Amazon Aurora. Auto Scaling no puede utilizarse para equilibrar la carga del tráfico HTTP y HTTPS.
**System Load Balancer** - Es una opción inventada y se ha añadido como distractor.
Referencia:
https://aws.amazon.com/elasticloadbalancing/

Pregunta 5: **Omitido¿Qué entidad de AWS te permite conectar de forma privada tu Amazon Virtual Private Cloud (Amazon VPC) a una cola de Amazon Simple Queue Service (Amazon SQS)?**
• **VPC Gateway Endpoint**
• **Internet Gateway**
• **VPC Interface Endpoint(Correcto)**
• **AWS Direct Connect
Explicación**
Opción correcta:
**VPC Interface Endpoint**
Un punto final de interfaz es una interfaz de red elástica (ENI) con una dirección IP privada del rango de direcciones IP de tu subred que sirve como punto de entrada para el tráfico destinado a un servicio soportado. Los puntos finales de interfaz funcionan con AWS PrivateLink, una tecnología que te permite acceder de forma privada a los servicios utilizando direcciones IP privadas. AWS PrivateLink restringe todo el tráfico de red entre tu VPC y los servicios a la red de Amazon. No necesitas Internet Gateway, un dispositivo de traducción de direcciones de red (NAT) ni una puerta de enlace virtual privada.
Alerta de examen:
Puede que veas una pregunta sobre este concepto en el examen. Recuerda que sólo Amazon S3 y Amazon DynamoDB soportan el punto final de gateway de VPC. Todos los demás servicios que admiten puntos finales de VPC utilizan un punto final de interfaz de VPC (ten en cuenta que Amazon S3 también admite el punto final de interfaz de VPC).
Opciones incorrectas:
**VPC Gateway Endpoint** - Un Gateway Endpoint es una puerta de enlace que especificas como destino de una ruta en tu tabla de rutas para el tráfico destinado a un servicio de AWS soportado. Se admiten los siguientes servicios de AWS: Amazon S3, DynamoDB. No puedes utilizar VPC Gateway Endpoint para conectar de forma privada tu VPC a una cola de Amazon SQS.
**AWS Direct Connect** - AWS Direct Connect es una solución de servicio en el Cloud que facilita el establecimiento de una conexión de red dedicada desde tus instalaciones a AWS. Puedes utilizar AWS Direct Connect para establecer una interfaz virtual privada desde tu red local directamente a tu Amazon VPC. Esta conexión privada tarda al menos un mes en completarse. No puedes utilizar AWS Direct Connect para conectar de forma privada tu VPC a una cola de Amazon SQS.
**Internet Gateway** - Internet Gateway es un componente de la VPC de escala horizontal, redundante y de alta disponibilidad que permite la comunicación entre tu VPC e Internet. Internet Gateway tiene dos finalidades: proporcionar un destino en las tablas de rutas de tu VPC para el tráfico enrutable por Internet, y realizar la traducción de direcciones de red (NAT) para las instancias a las que se han asignado direcciones IPv4 públicas. No puedes utilizar Internet Gateway para conectar de forma privada tu VPC a una cola de Amazon SQS.
Referencia:
https://docs.aws.amazon.com/vpc/latest/userguide/vpc-endpoints.html

Pregunta 6: **Omitido¿En qué se diferencia Amazon EC2 de los sistemas de alojamiento tradicionales? (Selecciona dos)**
• [ ] **Con Amazon EC2, los usuarios corren el riesgo de comprar recursos en exceso**
• [ ] **Amazon EC2 proporciona una instancia preconfigurada por un coste mensual fijo**
• [ ] **Amazon EC2 está más orientado a grupos de usuarios con requisitos de sistema similares, de modo que los recursos del servidor se comparten entre varios usuarios y se reduce el coste**
• [ ] **Con Amazon EC2, los desarrolladores pueden lanzar y terminar las instancias en cualquier momento que lo necesiten(Correcto)**
• [ ] **Amazon EC2 puede escalar con los cambios en los requisitos informáticos(Correcto)
Explicación**
Opciones correctas:
Amazon Elastic Compute Cloud (Amazon EC2) es un servicio web que proporciona capacidad informática segura y redimensionable en el Cloud con soporte para facturación por segundos. Es la forma más sencilla de aprovisionar servidores en el Cloud de AWS y acceder al sistema operativo subyacente.
Amazon EC2 se diferencia fundamentalmente de los sistemas tradicionales de alojamiento on-premise en la flexibilidad, el control y el importante ahorro de costes que ofrece a los desarrolladores, permitiéndoles tratar la instancia de Amazon EC2 como su propio servidor personalizado respaldado por la sólida infraestructura del Cloud de AWS.
**Amazon EC2 puede escalar con los cambios en los requisitos informáticos**
Cuando las necesidades informáticas cambian inesperadamente, Amazon EC2 puede escalarse para adaptarse a ellas. Los desarrolladores pueden controlar cuántas instancias EC2 están en uso en un momento dado.
**Con Amazon EC2, los desarrolladores pueden lanzar y terminar las instancias en cualquier momento que lo necesiten**
Utilizando Amazon EC2, los desarrolladores pueden elegir no sólo lanzar, terminar, iniciar o apagar instancias en cualquier momento, sino que también pueden personalizar completamente la configuración de sus instancias para adaptarlas a sus necesidades.
Opciones incorrectas:
**Amazon EC2 proporciona una instancia preconfigurada por un coste mensual fijo** - Esta opción es incorrecta. Los desarrolladores de EC2 disfrutan de la ventaja de pagar sólo por su consumo real de recursos, sin costes mensuales ni iniciales. Los desarrolladores pueden personalizar sus instancias EC2 para su pila de aplicaciones.
**Con Amazon EC2, los usuarios corren el riesgo de comprar recursos en exceso** - Esta afirmación es incorrecta. Los usuarios corren el riesgo de comprar de más en los servicios de alojamiento tradicionales, en los que los usuarios pagan una cuota fija por adelantado, independientemente de la potencia informática real que utilicen. Con EC2, los usuarios sólo pagan por los recursos realmente consumidos.
**Amazon EC2 está más orientado a grupos de usuarios con requisitos de sistema similares, de modo que los recursos del servidor se comparten entre varios usuarios y se reduce el coste** - Esta afirmación es incorrecta. Los recursos no se comparten entre usuarios en EC2, por lo que los usuarios tienen la flexibilidad de iniciar o apagar las instancias según sus necesidades. Esto no es posible en los sistemas de alojamiento tradicionales, en los que los recursos se comparten entre usuarios.
Referencia:
https://aws.amazon.com/ec2/faqs/

Pregunta 7: **OmitidoUna organización mantiene una Nube Privada Virtual (VPC) independiente para cada una de sus unidades de negocio. Dos unidades necesitan compartir datos de forma privada. ¿Cuál es la forma más óptima de compartir datos de forma privada entre las dos VPC?**
• **VPC Endpoint**
• **AWS Direct Connect**
• **VPC Peering(Correcto)**
• **VPN Site to Site
Explicación**
Opción correcta:
**VPC Peering**
Una conexión de VPC Peering es una conexión de red entre dos VPC que te permite enrutar el tráfico entre ellas de forma privada. Las instancias de cualquiera de las dos VPC pueden comunicarse entre sí como si estuvieran dentro de la misma red. Puedes crear una conexión de VPC Peering entre tus VPC, con una VPC en otra cuenta de AWS, o con una VPC en una región de AWS diferente.Visión general de la VPC Peering:  vía - https://docs.aws.amazon.com/vpc/latest/peering/what-is-vpc-peering.html
Opciones incorrectas:
**VPN Site to Site** - VPN Site to Site crea una conexión segura entre tu centro de datos o sucursal y tus recursos en el Cloud de AWS. Esta conexión va a través del Internet público. VPN Site to Site no puede utilizarse para interconectar VPC.
**AWS Direct Connect** - AWS Direct Connect crea una conexión privada dedicada desde una red distante a tu VPC. Se trata de una conexión privada que no utiliza Internet público. El establecimiento de esta conexión tarda al menos un mes. Direct Connect no puede utilizarse para interconectar VPC.
**VPC Endpoint** - VPC Endpoint (también llamado punto final de VPC) te permite conectar de forma privada tu VPC a los servicios de AWS compatibles y a los servicios de punto final de VPC impulsados por AWS PrivateLink sin necesidad de Internet Gateway, un dispositivo NAT, una conexión VPN o una conexión AWS Direct Connect. No puedes conectar dos VPC utilizando un punto final de VPC.
Referencia:
https://docs.aws.amazon.com/vpc/latest/peering/what-is-vpc-peering.html

Pregunta 8: **OmitidoUna organización multinacional tiene una nube privada virtual (VPC) independiente para cada una de sus unidades de negocio en el Cloud de AWS. La organización también quiere conectar su centro de datos local con las diferentes nubes privadas virtuales (VPC) para mejorar la colaboración en toda la organización. ¿Qué servicios de AWS se pueden combinar para crear la solución MÁS eficiente para este caso de uso? (Selecciona dos)**
• [ ] **AWS Direct Connect(Correcto)**
• [ ] **AWS Internet Gateway**
• [ ] **AWS Storage Gateway**
• [ ] **AWS Transit Gateway(Correcto)**
• [ ] **VPC Peering
Explicación**
Opción correcta:
**AWS Transit Gateway**
AWS Transit Gateway conecta Virtual Private Clouds (VPC) y redes locales a través de un hub central. Esto simplifica tu red y pone fin a las complejas relaciones de interconexión. Actúa como un router en el Cloud: cada nueva conexión sólo se realiza una vez. A medida que te expandes globalmente, el peering entre regiones conecta las puertas de enlace de tránsito de AWS utilizando la red global de AWS. Tus datos se cifran automáticamente y nunca viajan por el Internet público.Cómo AWS Transit Gateway puede simplificar tu red:  vía - https://aws.amazon.com/transit-gateway/
**AWS Direct Connect**
AWS Direct Connect es una solución de servicio en el Cloud que facilita el establecimiento de una conexión de red dedicada desde tus instalaciones a AWS. Con AWS Direct Connect, puedes establecer una conectividad privada entre AWS y tu centro de datos, oficina o entorno de colocación, lo que en muchos casos puede reducir tus costes de red, aumentar el rendimiento del ancho de banda y proporcionar una experiencia de red más consistente que las conexiones basadas en Internet.
Opciones incorrectas:
**VPC Peering** - Una conexión de VPC Peering es una conexión de red entre dos VPC que te permite enrutar el tráfico entre ellas de forma privada. La conexión de VPC Peering no es transitiva, hay que establecer una conexión de VPC Peering independiente entre dos VPC que necesiten hablar entre sí. Con un número creciente de nubes privadas virtuales (VPC), esto se hace difícil de gestionar.
No se permite la conexión de VPC Peering: vía - https://docs.aws.amazon.com/vpc/latest/peering/invalid-peering-configurations.html
**AWS Internet Gateway** - Internet Gateway es un componente de la VPC de escala horizontal, redundante y de alta disponibilidad, que permite la comunicación entre las instancias de tu VPC e Internet. Por tanto, no impone riesgos de disponibilidad ni limitaciones de ancho de banda a tu tráfico de red. No puedes utilizar la Gateway de Internet para conectar tu centro de datos local con varias VPC dentro de tu red de AWS.
**AWS Storage Gateway** - AWS Storage Gateway es un servicio híbrido de almacenamiento en el Cloud que te proporciona acceso local a un almacenamiento prácticamente ilimitado en el Cloud. Todos los datos transferidos entre la puerta de enlace y el almacenamiento de AWS se cifran mediante SSL (para los tres tipos de puertas de enlace: de archivos, de volumen y de cinta). No puedes utilizar AWS Storage Gateway para conectar tu centro de datos local con varias VPC dentro de tu red de AWS.
Referencia:
https://aws.amazon.com/transit-gateway/

Pregunta 9: **Omitido¿Cuáles de los siguientes son los servicios de computación sin servidor que ofrece AWS? (Selecciona dos)**
• [ ] **AWS Fargate(Correcto)**
• [ ] **AWS Elastic Beanstalk**
• [ ] **Amazon Elastic Compute Cloud (Amazon EC2)**
• [ ] **Amazon Lightsail**
• [ ] **AWS Lambda(Correcto)
Explicación**
Opciones correctas:
Sin servidor es la arquitectura nativa del Cloud que te permite trasladar más responsabilidades operativas a AWS, aumentando tu agilidad e innovación. Serverless te permite crear y ejecutar aplicaciones y servicios sin pensar en servidores. Elimina las tareas de gestión de la infraestructura, como el aprovisionamiento de servidores o clústeres, la aplicación de parches, el mantenimiento del sistema operativo y el aprovisionamiento de capacidad.Visión general de la plataforma sin servidor de AWS:  vía - https://aws.amazon.com/serverless/
**AWS Lambda**
Con AWS Lambda, puedes ejecutar código para prácticamente cualquier tipo de aplicación o servicio backend, todo ello con cero administración. Sólo tienes que subir tu código y Lambda se encarga de todo lo necesario para ejecutar y escalar tu código con alta disponibilidad. Puedes configurar tu código para que se active automáticamente desde otros servicios de AWS o llamarlo directamente desde cualquier aplicación web o móvil.
AWS Lambda te permite ejecutar código sin aprovisionar ni administrar servidores. Sólo pagas por el tiempo de computación que consumes: no hay cargos cuando tu código no se está ejecutando.
**AWS Fargate**
AWS Fargate es un motor informático sin servidor para contenedores que funciona tanto con Amazon Elastic Container Service (ECS) como con Amazon Elastic Kubernetes Service (EKS). Fargate facilita que te centres en crear tus aplicaciones. Fargate elimina la necesidad de aprovisionar y administrar servidores, te permite especificar y pagar recursos por aplicación, y mejora la seguridad mediante el aislamiento de aplicaciones por diseño.
AWS Fargate es un motor informático sin servidor diseñado específicamente para contenedores. Fargate escala y administra la infraestructura necesaria para ejecutar tus contenedores.
Opciones incorrectas:
**Amazon Elastic Compute Cloud (Amazon EC2)** - Amazon Elastic Compute Cloud (Amazon EC2) es un servicio web que proporciona capacidad informática segura y redimensionable en el Cloud con soporte para facturación por segundos. Es la forma más sencilla de aprovisionar servidores en el Cloud de AWS y acceder al sistema operativo subyacente. Amazon EC2 reduce a minutos el tiempo necesario para obtener y arrancar nuevas instancias de servidor, lo que te permite escalar rápidamente la capacidad, tanto hacia arriba como hacia abajo, a medida que cambian tus necesidades informáticas.
**AWS Elastic Beanstalk** - AWS Elastic Beanstalk es un servicio fácil de usar para implementar y escalar aplicaciones y servicios web. Sólo tienes que subir tu código y Elastic Beanstalk se encarga automáticamente del despliegue, desde el aprovisionamiento de capacidad, el equilibrio de carga y el autoescalado hasta la monitorización del estado de la aplicación. Beanstalk aprovisiona servidores, por lo que no es un servicio sin servidor.
**Amazon Lightsail** - Amazon Lightsail es una plataforma en el Cloud fácil de usar que te ofrece todo lo necesario para crear una aplicación o un sitio web, además de un plan mensual rentable. Lightsail ofrece varios sistemas operativos, pilas de desarrollo y aplicaciones web preconfigurados y listos para usar con un solo clic, como Linux, Windows OS y WordPress.
Referencias:
https://aws.amazon.com/serverless/

Pregunta 13: **OmitidoUna empresa de transporte de mercancías ejecuta su flota de servidores en instancias de Amazon Elastic Compute Cloud (Amazon EC2). Algunas de estas instancias alojan las aplicaciones CRM (Customer Relationship Management) que necesitan estar accesibles 24*7. Estas aplicaciones no son de misión crítica. En caso de desastre, estas aplicaciones pueden gestionarse en un número menor de instancias durante algún tiempo.
¿Qué estrategia de recuperación en caso de catástrofe es adecuada y rentable para este requisito?**
• **Estrategia de copia de seguridad y restauración**
• **Estrategia de luz piloto**
• **Estrategia multisitio activo-activo**
• **Estrategia de espera en caliente(Correcto)
Explicación**
Opción correcta:
**Estrategia de espera en caliente**
Al seleccionar tu estrategia de recuperación ante desastres (DR), debes sopesar los beneficios de un menor RTO (objetivo de tiempo de recuperación) y RPO (objetivo de punto de recuperación) frente a los costes de implantación y funcionamiento de una estrategia. Tanto la estrategia de piloto como la de espera caliente ofrecen un buen equilibrio entre beneficios y costes.
Esta estrategia replica los datos de la región primaria a los recursos de datos de la región de recuperación, como instancias de base de datos relacionales de Amazon (Amazon RDS) o tablas de Amazon DynamoDB. Estos recursos de datos están listos para servir solicitudes. Además de la replicación, esta estrategia requiere que crees una copia de seguridad continua en la región de recuperación. Esto se debe a que cuando se producen desastres del tipo "acción humana", los datos pueden borrarse o corromperse, y la replicación replicará los datos defectuosos. Las copias de seguridad son necesarias para que puedas volver al último estado bueno conocido.
La estrategia de espera en caliente despliega una pila funcional, pero a capacidad reducida. El punto final de RD puede gestionar solicitudes, pero no puede gestionar los niveles de tráfico de producción. Puede ser mayor, pero siempre es menor que el despliegue de producción completa para ahorrar costes. Sin embargo, si la pila pasiva se despliega en la región de recuperación a plena capacidad, esta estrategia se conoce como "espera en caliente". Como la espera en caliente despliega una pila funcional en la región de recuperación, esto facilita la comprobación de la preparación de la región mediante transacciones sintéticas.Estrategias de recuperación ante catástrofes (RD):  vía - https://aws.amazon.com/blogs/architecture/disaster-recovery-dr-architecture-on-aws-part-iii-pilot-light-and-warm-standby/
Opciones incorrectas:
**Estrategia multisitio activo-activo** - Esta estrategia utiliza regiones de AWS como tus sitios activos, creando una arquitectura activa/activa multirregión. Por lo general, se utilizan dos regiones. Cada región aloja una pila de cargas de trabajo multizona de disponibilidad (AZ) de alta disponibilidad. En cada región, los datos se replican en directo entre los almacenes de datos y también se realizan copias de seguridad. Esto protege contra desastres que incluyan la eliminación o corrupción de datos, ya que la copia de seguridad de los datos puede restaurarse al último estado bueno conocido. Cada pila regional sirve eficazmente al tráfico de producción. Pero esta estrategia implica costes y sólo debe utilizarse para aplicaciones de misión crítica.
**Estrategia de luz piloto** - Al igual que la estrategia espera en caliente, replica los datos de la región primaria a los recursos de datos de la región de recuperación, como las instancias de base de datos de Amazon Relational Database Service (Amazon RDS) o las tablas de Amazon DynamoDB. Pero, la región de DR en una estrategia de luz piloto (a diferencia de la de espera en caliente) no puede servir solicitudes hasta que se tomen medidas adicionales. Una luz piloto en una caldera doméstica no proporciona calor a la casa. Proporciona una forma rápida de encender los quemadores del horno que luego proporcionan calor.
La espera en caliente puede gestionar el tráfico a niveles reducidos inmediatamente. La luz piloto requiere que primero despliegues la infraestructura y luego escales los recursos antes de que la carga de trabajo pueda gestionar las peticiones.
**Estrategia de copia de seguridad y restauración** - La copia de seguridad y la restauración se asocian a un mayor RTO (objetivo de tiempo de recuperación) y RPO (objetivo de punto de recuperación). Esto se traduce en tiempos de inactividad más largos y una mayor pérdida de datos entre el momento en que se produce el evento de desastre y la recuperación. Sin embargo, la copia de seguridad y restauración puede seguir siendo la estrategia adecuada para las cargas de trabajo, porque es la más fácil y menos costosa de aplicar.
Referencia:
https://aws.amazon.com/blogs/architecture/disaster-recovery-dr-architecture-on-aws-part-iii-pilot-light-and-warm-standby/

Pregunta 37: **OmitidoUna empresa de medios de comunicación social quiere tener la estrategia MÁS rentable para desplegar instancias de Amazon Elastic Compute Cloud (Amazon EC2). Como Cloud Practitioner, ¿cuál de las siguientes opciones recomendarías? (Selecciona dos)**
• [ ] **Utiliza instancias reservadas (RI) para ejecutar aplicaciones con un uso predecible durante el próximo año(Correcto)**

• [ ] **Utilizar instancias bajo demanda para trabajos ad hoc que puedan interrumpirse**
• [ ] **Utiliza instancias bajo demanda para ejecutar aplicaciones con un uso predecible durante el próximo año**
• [ ] **Utiliza instancias reservadas (RI) para trabajos ad hoc que puedan interrumpirse
Explicación**
Opciones correctas:
**Utilizar instancias spot para trabajos ad-hoc que puedan interrumpirse**
Una instancia de spot es una instancia EC2 no utilizada que está disponible por menos del precio bajo demanda. Como las instancias spot te permiten solicitar instancias EC2 no utilizadas con grandes descuentos (hasta el 90%), puedes reducir significativamente tus costes de Amazon EC2. Las instancias spot son muy adecuadas para análisis de datos, trabajos por lotes, procesamiento en segundo plano y tareas opcionales. Pueden interrumpirse con poca antelación, por lo que no son adecuadas para cargas de trabajo críticas que deban ejecutarse en un momento determinado.
**Utiliza instancias reservadas (RI) para ejecutar aplicaciones con un uso predecible durante el próximo año**
Las instancias reservadas (RI) te proporcionan un ahorro significativo (hasta el 75%) en tus costes de Amazon EC2 en comparación con los precios de las instancias bajo demanda. Las instancias reservadas (RI) no son instancias físicas, sino un descuento de facturación aplicado al uso de instancias bajo demanda en tu cuenta. Puedes adquirir una instancia reservada (RI) para un compromiso de uno o tres años, siendo el compromiso de tres años el que ofrece un mayor descuento. Las instancias reservadas (RI) son ideales para aplicaciones con un uso constante. Las instancias reservadas (RI) no pueden interrumpirse.Visión general de las opciones de precios de Amazon EC2:  vía - https://aws.amazon.com/ec2/pricing/
Opciones incorrectas:
**Utiliza instancias bajo demanda para ejecutar aplicaciones con un uso predecible durante el próximo año**
**Utilizar instancias bajo demanda para trabajos ad hoc que puedan interrumpirse**
Una instancia bajo demanda es una instancia que utilizas bajo demanda. Tienes pleno control sobre su ciclo de vida: tú decides cuándo lanzarla, detenerla, hibernarla, iniciarla, reiniciarla o terminarla. Pregunta 64: **Omitido¿Cuáles de las siguientes entidades forman parte de una Amazon Virtual Private Cloud (Amazon VPC) en el Cloud de AWS? (Selecciona dos)**
• [ ] **Subred(Correcto)**
• [ ] **API Gateway**
• [ ] **Internet Gateway(Correcto)**
• [ ] **Storage Gateway**
• [ ] **Objeto
Explicación**
Opciones correctas:
**Subred**
**Internet Gateway**
Amazon Virtual Private Cloud (Amazon VPC) te permite lanzar recursos de AWS en una red virtual que hayas definido.
Los siguientes son los conceptos clave de las VPC:
Virtual Private Cloud (VPC) - Una red virtual dedicada a tu cuenta de AWS.
Subred - Un rango de direcciones IP en tu VPC.
Tabla de enrutamiento (tabla de rutas) - Conjunto de reglas, llamadas rutas, que se utilizan para determinar hacia dónde se dirige el tráfico de la red.
Gateway de Internet - Una puerta de enlace que adjuntas a tu VPC para permitir la comunicación entre los recursos de tu VPC e Internet.
VPC endpoint - Te permite conectar de forma privada tu VPC a los servicios de AWS compatibles y a los servicios de VPC endpoint impulsados por PrivateLink sin necesidad de Internet Gateway, un dispositivo NAT, una conexión VPN o una conexión AWS Direct Connect.
Opciones incorrectas:
**Storage Gateway** - AWS Storage Gateway es un servicio de almacenamiento en el Cloud híbrido que te proporciona acceso local a un almacenamiento en el Cloud prácticamente ilimitado. Los clientes utilizan Storage Gateway para simplificar la gestión del almacenamiento y reducir los costes en casos clave de uso del almacenamiento en el Cloud híbrido. Storage Gateway no forma parte de VPC.
**API Gateway** - Amazon API Gateway es un servicio totalmente administrado que facilita a los desarrolladores la creación, publicación, mantenimiento, monitorización y seguridad de API a cualquier escala. Las API actúan como la "puerta de entrada" para que las aplicaciones accedan a los datos, la lógica empresarial o la funcionalidad de tus servicios backend. API Gateway no forma parte de una VPC.
**Objeto** - Los buckets y los objetos forman parte de Amazon S3. Amazon Simple Storage Service (Amazon S3) es un servicio de almacenamiento de objetos que ofrece escalabilidad, disponibilidad de datos, seguridad y rendimiento líderes en el sector.
Referencia:
https://docs.amazonaws.cn/en_us/vpc/latest/userguide/what-is-amazon-vpc.html

Pregunta 65: **OmitidoAWS Shield Advanced proporciona protección ampliada contra ataques DDoS para las aplicaciones web que se ejecutan en ¿cuál de los siguientes recursos? (Selecciona dos)**
• [ ] **Amazon CloudFront(Correcto)**
• [ ] **AWS Identity and Access Management (IAM)**
• [ ] **AWS Elastic Beanstalk**
• [ ] **Amazon Elastic Compute Cloud (Amazon EC2)(Correcto)**
• [ ] **Amazon Simple Storage Service (Amazon S3)
Explicación**
Opciones correctas:
**Amazon CloudFront**
**Amazon Elastic Compute Cloud (Amazon EC2)**
AWS Shield Standard está activado para todos los clientes de AWS, por defecto. Para niveles superiores de protección contra ataques, puedes suscribirte a AWS Shield Advanced. Con Shield Advanced, también tienes acceso exclusivo a métricas e informes avanzados en tiempo real para obtener una amplia visibilidad de los ataques a tus recursos de AWS. Con la ayuda del DRT (equipo de respuesta DDoS), AWS Shield Advanced incluye detección y mitigación inteligente de ataques DDoS no sólo para los ataques de la capa de red (capa 3) y de transporte (capa 4), sino también para los de la capa de aplicación (capa 7).
AWS Shield Advanced proporciona protección ampliada contra ataques DDoS para aplicaciones web que se ejecutan en los siguientes recursos: Amazon Elastic Compute Cloud, Elastic Load Balancer (ELB), Amazon CloudFront, Amazon Route 53, AWS Global Accelerator.
Opciones incorrectas:
**Amazon Simple Storage Service (Amazon S3)**
**AWS Elastic Beanstalk**
**AWS Identity and Access Management (IAM)**
Estos tres tipos de recursos no son compatibles con AWS Shield Advanced.
Referencia:
https://docs.aws.amazon.com/waf/latest/developerguide/ddos-overview.html

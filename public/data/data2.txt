Pregunta 1: **OmitidoUna empresa está migrando su base de datos local a Amazon RDS. ¿Qué debe hacer la empresa para garantizar que los costos de Amazon RDS se mantengan al mínimo?**
• **Combine capacidad de instancias reservadas bajo demanda con planes de ahorro**
• **Usar una arquitectura Active-Active de varias regiones**
• **Usar una arquitectura activo-pasivo de varias regiones**
• **Tamaño correcto antes y después de la migración (Right-size before and after migration)(Correcto)
Explicación**
El tamaño correcto es el proceso de hacer coincidir los tipos y tamaños de instancia con el rendimiento de la carga de trabajo y los requisitos de capacidad al menor costo posible. Al ajustar el tamaño antes de la migración, puede reducir significativamente los costos de infraestructura. Si omite el tamaño correcto para ahorrar tiempo, su velocidad de migración podría ser más rápida, pero terminará con un mayor gasto en infraestructura en la nube durante un tiempo potencialmente largo.
Debido a que sus necesidades de recursos siempre están cambiando, el tamaño correcto debe convertirse en un proceso continuo para lograr continuamente la optimización de costos. Es importante ajustar el tamaño cuando considere pasar a la nube por primera vez y calcular el costo total de propiedad. Sin embargo, es igualmente importante ajustar el tamaño periódicamente una vez que esté en la nube para garantizar una optimización continua de los costos y el rendimiento.
Elegir una instancia de Amazon RDS para una carga de trabajo determinada significa encontrar la familia de instancias que más se acerque a las necesidades de CPU, E/S de disco y memoria de su carga de trabajo. Amazon RDS proporciona una amplia selección de instancias, lo que le brinda mucha flexibilidad para ajustar el tamaño de sus recursos para satisfacer las necesidades de capacidad al menor costo.

***Las otras opciones son incorrectas:***

***"Usar una arquitectura activo-activa de varias regiones" es incorrecto.*** Con la solución Multi-Region Active-Active, su carga de trabajo se implementa en varias regiones de AWS y atiende activamente el tráfico de varias regiones. Los clientes de AWS utilizan este enfoque para reducir la latencia de los usuarios globales y lograr el nivel más alto de disponibilidad. El uso de una arquitectura Active-Active de varias regiones aumentará los costos de infraestructura, incluidos los costos de Amazon RDS.

***"Usar una arquitectura activo-pasiva multirregión" es incorrecto.*** Con la arquitectura activo-pasivo de varias regiones, la carga de trabajo se implementa en dos regiones de AWS (una región principal y una región en espera). En esta arquitectura, las solicitudes de usuario se atienden solo desde la región principal. Si la región principal se cae debido a un desastre natural o cualquier otra razón, la otra región seguirá estando disponible y atenderá las solicitudes de los usuarios. Los clientes de AWS utilizan este enfoque con fines de recuperación ante desastres. El uso de una arquitectura activo-pasiva de varias regiones aumentará los costos de infraestructura, incluidos los costos de Amazon RDS.

***"Combinar reservas de capacidad bajo demanda con planes de ahorro" es incorrecto.*** Cuando combina reservas de capacidad bajo demanda con planes de ahorro, podrá reducir los costos significativamente. Sin embargo, las reservas de capacidad bajo demanda solo están disponibles para Amazon EC2. Para obtener más información sobre las reservas de capacidad bajo demanda, consulte este enlace: https://docs.aws.amazon.com/AWSEC2/latest/UserGuide/instance-purchasing-options.html

Pregunta 2: **Omitido¿Dónde puede almacenar archivos en AWS? (Elija DOS)**
• [ ] **Amazon ECS**
• [ ] **Amazon EBS(Correcto)**
• [ ] **Amazon SNS**
• [ ] **Amazon EMR**
• [ ] **Amazon EFS(Correcto)
Explicación**
**Amazon Elastic File System (Amazon EFS)** proporciona almacenamiento de archivos simple, escalable y elástico para su uso con los servicios en la nube de AWS y los recursos locales. Es fácil de usar y ofrece una interfaz simple que le permite crear y configurar sistemas de archivos de forma rápida y sencilla. Amazon EFS está diseñado para escalar elásticamente bajo demanda sin interrumpir las aplicaciones, creciendo y reduciéndose automáticamente a medida que agrega y elimina archivos, de modo que sus aplicaciones tengan el almacenamiento que necesitan, cuando lo necesitan. Está diseñado para proporcionar acceso compartido paralelo masivo a miles de instancias de Amazon EC2, lo que permite a sus aplicaciones alcanzar altos niveles de rendimiento agregado e IOPS que se escalan a medida que crece un sistema de archivos, con latencias bajas consistentes. Como servicio regional, Amazon EFS está diseñado para una alta disponibilidad y durabilidad, almacenando datos de forma redundante en varias zonas de disponibilidad.
**Amazon Elastic Block Store (Amazon EBS)** proporciona volúmenes de almacenamiento en bloque persistentes para su uso con instancias Amazon EC2 en la nube de AWS. Cada volumen de Amazon EBS se replica automáticamente dentro de su zona de disponibilidad para protegerlo de fallos en los componentes, lo que ofrece alta disponibilidad y durabilidad.

***Las otras opciones son incorrectas:***

***"Amazon SNS" es incorrecto.*** Amazon Simple Notification Service (SNS) es un servicio de mensajería pub/sub.

***"Amazon ECS" es incorrecto.*** Amazon Elastic Container Service (ECS) es un servicio informático que se utiliza para ejecutar aplicaciones en contenedores en AWS.

***"Amazon EMR" es incorrecto.*** Amazon Elastic MapReduce (EMR) es un servicio web que permite a las empresas, investigadores, analistas de datos y desarrolladores procesar de forma fácil y rentable grandes cantidades de datos en instancias de Amazon EC2 escalables dinámicamente.

Pregunta 4: **Omitido¿Cuál de los siguientes procedimientos le ayudará a reducir los costos de Amazon S3?**
• **Utilice la combinación correcta de clases de almacenamiento basadas en diferentes casos de uso(Correcto)**
• **Elija la zona de disponibilidad adecuada para su bucket de S3**
• **Utilice la característica Importar/Exportar para mover archivos antiguos automáticamente a Amazon Glacier**
• **Mueva todos los datos almacenados en el estándar S3 a EBS
Explicación**
**Amazon S3** ofrece una amplia gama de clases de almacenamiento diseñadas para diferentes casos de uso. Estos incluyen el estándar S3 para el almacenamiento de propósito general de datos a los que se accede con frecuencia; S3 Intelligent-Tiering para datos con patrones de acceso desconocidos o cambiantes; Acceso estándar poco frecuente de S3 (S3 Standard-IA) y acceso infrecuente de una zona de S3 (S3 One Zone-IA) para datos de larga duración, pero a los que se accede con menos frecuencia; y Amazon S3 Glacier (S3 Glacier) y Amazon S3 Glacier Deep Archive (S3 Glacier Deep Archive) para el archivo a largo plazo y la preservación digital.

***Las otras opciones son incorrectas:***

***"Mover todos los datos almacenados en el estándar S3 a EBS" es incorrecto.*** EBS es un servicio de almacenamiento a nivel de bloque que es significativamente más caro que S3.

***"Elija la zona de disponibilidad correcta para su bucket de S3" es incorrecto.*** No tiene la opción de almacenar objetos en una zona de disponibilidad específica. Solo puede elegir la región de AWS en la que desea que residan los buckets de S3.
Información adicional:
En general, los precios de AWS para un recurso pueden cambiar en función de la región de AWS donde se crea, NO en función de las zonas de disponibilidad dentro de la misma región.

***"Utilice la característica Importar/Exportar para mover archivos antiguos automáticamente a Amazon Glacier" es incorrecto.*** Mover datos antiguos automáticamente a Amazon Glacier ayudará a reducir los costos de Amazon S3, pero esto se puede hacer utilizando las políticas del ciclo de vida de Amazon S3, NO la característica Importar/Exportar. Para reducir los costos de Amazon S3, debe crear una política de ciclo de vida para mover automáticamente archivos antiguos (o a los que se accede con poca frecuencia) a niveles de almacenamiento menos costosos, o para eliminarlos automáticamente después de una duración especificada.

Pregunta 5: **Omitido¿Cuál de las siguientes opciones puede ayudar a proteger sus instancias EC2 de ataques DDoS? (Elija DOS)**
• [ ] **AWS Batch**
• [ ] **AWS IAM**
• [ ] **AWS CloudHSM**
• [ ] **Security Groups (Grupos de seguridad)(Correcto)**
• [ ] **Listas de control de acceso a la red (ACL de red) Network Access Control Lists (Network ACLs)(Correcto)
Explicación**
Los actores malintencionados a veces utilizan ataques distribuidos de denegación de servicio (DDoS) para inundar una red, sistema o aplicación con más tráfico, conexiones o solicitudes de las que puede manejar.
Cuando se trata de ataques DDoS, es importante minimizar las oportunidades que tiene un atacante para atacar sus aplicaciones. Esto significa restringir el tipo de tráfico que puede llegar a sus aplicaciones. La configuración de grupos de seguridad y ACL de red en Amazon VPC es una herramienta eficaz para ayudar a filtrar el tráfico y reducir la superficie de ataque de sus aplicaciones.
Los grupos de seguridad le permiten controlar el tráfico entrante y saliente a sus instancias Amazon EC2 al permitir específicamente la comunicación solo en los puertos y protocolos necesarios para sus aplicaciones. El acceso a cualquier otro puerto o protocolo se deniega automáticamente.
Las ACL de red proporcionan una capa adicional de defensa para su VPC al permitirle crear reglas de permitir y denegar que se procesan en orden numérico, al igual que un firewall tradicional. Esto es útil para permitir o denegar el tráfico a nivel de subred, a diferencia de los grupos de seguridad que filtran el tráfico a nivel de instancia EC2. Por ejemplo, si ha identificado direcciones IP de Internet o rangos que no son deseados o potencialmente abusivos, puede impedir que lleguen a su aplicación con una regla de denegación de ACL de red.

**Información adicional:**
AWS no configura grupos de seguridad ni ACL de red para protegerle de los ataques DDoS. Es responsabilidad del cliente establecer las reglas adecuadas de ACL de red y grupo de seguridad para protegerse de estos ataques y asegurar su red.
Además de los grupos de seguridad y las ACL de red, AWS proporciona infraestructura y servicios flexibles que ayudan a los clientes a implementar mitigaciones de DDoS sólidas y crear arquitecturas de aplicaciones de alta disponibilidad que siguen las prácticas recomendadas de AWS para la resiliencia de DDoS. Estos incluyen servicios como **Amazon Route 53, Amazon CloudFront, Elastic Load Balancing y AWS WAF**para controlar y absorber el tráfico y desviar las solicitudes no deseadas. Estos servicios se integran con **AWS Shield**, un servicio de protección DDoS administrado que proporciona detección siempre activa y mitigaciones automáticas en línea para proteger las aplicaciones web que se ejecutan en AWS.

***Las otras opciones son incorrectas:***

***"AWS CloudHSM" es incorrecto.*** AWS CloudHSM es un módulo de seguridad de hardware (HSM) basado en la nube que le permite generar y utilizar fácilmente sus propias claves de cifrado en la nube de AWS.

***"AWS IAM" es incorrecto.*** AWS IAM le permite administrar el acceso a los servicios y recursos de AWS de forma segura.

***"AWS Batch" es incorrecto.*** AWS Batch es un servicio informático que le permite ejecutar cientos de miles de trabajos informáticos por lotes en AWS.

**Referencias:**
https://aws.amazon.com/answers/networking/aws-ddos-attack-mitigation/

Pregunta 10: **Omitido¿Qué servicios de AWS se pueden utilizar para mejorar el rendimiento de una aplicación global y reducir la latencia para sus usuarios? (Elija DOS)**
• [ ] **AWS Direct Connect**
• [ ] **AWS KMS**
• [ ] **AWS Global accelerator(Correcto)**
• [ ] **Amazon CloudFront(Correcto)**
• [ ] **AWS Glue
Explicación**
**AWS Global Accelerator y CloudFront** son dos servicios independientes que utilizan la red global de AWS y sus ubicaciones de borde en todo el mundo. Amazon CloudFront mejora el rendimiento de las aplicaciones globales al almacenar en caché el contenido en la ubicación de borde más cercana a los usuarios finales. AWS Global Accelerator mejora el rendimiento de las aplicaciones globales al enrutar las solicitudes de los usuarios finales a la región de AWS más cercana. Amazon CloudFront mejora el rendimiento tanto para el contenido almacenable en caché (por ejemplo, imágenes y vídeos) como para el dinámico (por ejemplo, la entrega dinámica de sitios). Global Accelerator es una buena opción para casos de uso específicos, como juegos, IoT o voz sobre IP.

Nota: AWS Global Accelerator no almacena en caché el contenido en ubicaciones de borde como Amazon CloudFront. AWS Global Accelerator utiliza las ubicaciones de borde de AWS para recibir solicitudes de usuarios finales y, a continuación, enruta estas solicitudes a la región de AWS más cercana a través de la red global de AWS.

***Las otras opciones son incorrectas:***

***"AWS KMS" es incorrecto.*** AWS KMS es un servicio de administración de claves que le facilita la creación y administración de claves de cifrado y el control de su uso en una amplia gama de servicios de AWS y en sus aplicaciones.

***"AWS Direct Connect" es incorrecto.*** AWS Direct Connect es una solución de servicio en la nube que se utiliza para establecer una conexión de red dedicada desde sus instalaciones a AWS.

***"AWS Glue" es incorrecto.*** AWS Glue es un servicio totalmente administrado, de extracción, transformación y carga (ETL) que automatiza los pasos lentos de la preparación de datos para el análisis.
Extraer, transformar y cargar (ETL) es el proceso de extraer (recopilar) datos de varias fuentes (de diferentes bases de datos, por ejemplo), **transformar** los datos en función de las reglas / necesidades comerciales (este paso ayuda a preparar los datos para el análisis y la toma de decisiones) y **cargar** los datos en una base de datos de destino, a menudo un almacén de datos.

Si los componentes de una aplicación están estrechamente acoplados (interconectados) y falla un componente, también fallará toda la aplicación. **Amazon SQS** y **Amazon SNS** son herramientas eficaces que le ayudan a crear aplicaciones de acoplamiento flexible. Comprender cómo funcionan los servicios SQS y SNS no es necesario para el nivel de Cloud Practitioner, pero tomemos un ejemplo simple, digamos que tiene dos componentes en su aplicación, el componente A y el componente B. El componente A envía mensajes (trabajos) al componente B para procesar. Ahora, ¿qué sucede si el componente A envía una gran cantidad de mensajes al mismo tiempo? El componente B fallará y toda la aplicación fallará. SQS actúa como intermediario, recibe y almacena mensajes del componente A, y el componente B extrae y procesa mensajes a su propio ritmo. De esta manera, ambos componentes se ejecutan independientemente el uno del otro.

***"Usar políticas de IAM para mantener el rendimiento" es incorrecto.*** No existe ninguna relación entre las políticas de IAM y el rendimiento. Las políticas de IAM se utilizan para conceder a los usuarios permiso para realizar acciones específicas en AWS.

Información adicional:
AWS proporciona acceso físico al centro de datos solo a los empleados aprobados. Todos los empleados que necesitan acceso al centro de datos primero deben solicitar acceso y proporcionar una justificación comercial válida. Estas solicitudes se conceden en función del principio de privilegios mínimos, donde las solicitudes deben especificar a qué capa del centro de datos necesita acceso el individuo y están limitadas en el tiempo. Las solicitudes son revisadas y aprobadas por personal autorizado, y el acceso se revoca después de que expire el tiempo solicitado.
El principio de privilegios mínimos no sólo se aplica al control del acceso físico. Los clientes de AWS también deben aplicar este principio al conceder permisos a los usuarios de IAM. En otras palabras, los clientes de AWS deben conceder a los usuarios de IAM solo los permisos que necesitan para realizar una tarea y nada más.

Pregunta 27: **Omitido¿Cuál de las siguientes actividades puede ayudar a reducir los costos mensuales de AWS? (Elija DOS)**
• [ ] **Implementación de sus recursos de AWS en varias zonas de disponibilidad**
• [ ] **Eliminación de todas las etiquetas de asignación de costos**
• [ ] **Creación de una política de ciclo de vida para mover datos a los que se accede con poca frecuencia a niveles de almacenamiento de información menos costosos(Correcto)**
• [ ] **Uso de AWS Network Load Balancer (NLB) para equilibrar la carga de las solicitudes HTTP entrantes**
• [ ] **Habilitación de Amazon EC2 Auto Scaling para todas sus cargas de trabajo(Correcto)
Explicación**
Amazon EC2 Auto Scaling supervisa sus aplicaciones y ajusta automáticamente la capacidad (hacia arriba o hacia abajo) para mantener un rendimiento estable y predecible al menor costo posible. Cuando disminuya la demanda, Amazon EC2 Auto Scaling eliminará automáticamente cualquier exceso de capacidad para evitar gastos excesivos. Cuando aumenta la demanda, Amazon EC2 Auto Scaling agregará automáticamente capacidad para mantener el rendimiento.
Para Amazon S3 y Amazon EFS, puede crear una política de ciclo de vida para mover automáticamente los datos a los que se accede con poca frecuencia a niveles de almacenamiento menos costosos. Para reducir los costos de Amazon S3, debe crear una política de ciclo de vida para mover automáticamente archivos antiguos (o a los que se accede con poca frecuencia) a niveles de almacenamiento menos costosos, como Amazon Glacier, o para eliminarlos automáticamente después de una duración especificada. Del mismo modo, puede crear una política de ciclo de vida de Amazon EFS para mover automáticamente los datos a los que se accede con menos frecuencia a niveles de almacenamiento menos costosos, como Amazon EFS Standard-Infrequent Access (EFS Standard-IA) y Amazon EFS One Zone-Infrequent Access (EFS One Zone-IA). Las clases de almacenamiento de acceso poco frecuente de Amazon EFS proporcionan un precio/rendimiento optimizado para los archivos a los que no se accede todos los días, con precios **de almacenamiento hasta un 92 % más** bajos en comparación con las clases de almacenamiento Amazon EFS Standard (EFS Standard) y Amazon EFS One Zone (EFS One Zone), respectivamente.

***Las otras opciones son incorrectas:***

***"Eliminar todas las etiquetas de asignación de costos" es incorrecto.*** Una etiqueta es una etiqueta que usted o AWS asignan a un recurso de AWS. Cada etiqueta consta de una *clave* y un *valor*. Para cada recurso, cada clave de etiqueta debe ser única y cada clave de etiqueta solo puede tener un valor. Puedes usar etiquetas para organizar tus recursos (por proyecto, equipo, ... etc.), y etiquetas de asignación de costos para realizar un seguimiento de los costos de AWS a un nivel detallado. Después de activar las etiquetas de asignación de costos, AWS utiliza las etiquetas de asignación de costos para organizar los costos de recursos en el informe de asignación de costos, a fin de facilitar la categorización y el seguimiento de los costos de AWS. La eliminación de todas las etiquetas de asignación de costos no ayudará a reducir los costos mensuales de AWS.

***"Implementación de los recursos de AWS en varias zonas de disponibilidad" es incorrecto.*** La implementación de sus recursos de AWS en varias zonas de disponibilidad puede ayudar a aumentar la disponibilidad y la tolerancia a errores de su aplicación.

***"Uso de AWS Network Load Balancer (NLB) para equilibrar la carga de las solicitudes HTTP entrantes" es incorrecto.*** Esta opción es incorrecta por dos razones:
1º: El equilibrio de carga no reduce los costos, Elastic Load Balancing distribuye automáticamente el tráfico entrante de aplicaciones de manera uniforme entre varios destinos, como instancias Amazon EC2, contenedores y funciones de Lambda, y le ayuda a obtener un rendimiento de aplicaciones más consistente.
2º: El equilibrador de carga recomendado para el tráfico HTTP es el AWS Application Load Balancer, NO el AWS Network Load Balancer.
**Información adicional:**
Elastic Load Balancing admite cuatro tipos de balanceadores de carga (Balanceador de carga de aplicaciones, Equilibrador de carga de red, Equilibrador de carga de puerta de enlace y Equilibrador de carga clásico). Puede seleccionar el equilibrador de carga adecuado en función de las necesidades de la aplicación.
1- Si necesita equilibrar la carga de las solicitudes HTTP\HTTPS, AWS recomienda utilizar el Balanceador de carga de aplicaciones.
2- Para el equilibrio de carga de protocolos de red/transporte (capa 4 – TCP, UDP) y para aplicaciones de rendimiento extremo/baja latencia, AWS recomienda utilizar Network Load Balancer.
3- Para administrar y distribuir el tráfico entre varios dispositivos virtuales de terceros, AWS recomienda utilizar el equilibrador de carga de gateway.
4- Si tiene una aplicación existente construida dentro de la red EC2-Classic, debe utilizar un Classic Load Balancer.

Pregunta 57: **OmitidoQué declaración describe mejor el pilar de excelencia operativa de AWS Well-Architected Framework?**
• **El uso eficiente de los recursos informáticos para cumplir con los requisitos**
• **La capacidad de monitorear sistemas y mejorar los procesos y procedimientos de soporte(Correcto)**
• **La capacidad de administrar las operaciones del centro de datos de manera más eficiente**
• **La capacidad de un sistema para recuperarse con agilidad de una falla
Explicación**
Los 5 pilares del marco  **AWS Well-Architected Framework**:
1- Excelencia operativa: El pilar de excelencia operativa incluye la capacidad de ejecutar y monitorear sistemas para ofrecer valor comercial y mejorar continuamente los procesos y procedimientos de soporte.
2- Seguridad: El pilar de seguridad incluye la capacidad de proteger la información, los sistemas y los activos al tiempo que ofrece valor comercial a través de evaluaciones de riesgos y estrategias de mitigación.
3- Confiabilidad: El pilar de confiabilidad incluye la capacidad de un sistema para recuperarse de interrupciones de infraestructura o servicio, adquirir dinámicamente recursos informáticos para satisfacer la demanda y mitigar interrupciones como configuraciones erróneas o problemas transitorios de red.
4- Eficiencia del rendimiento: El pilar de eficiencia del rendimiento incluye la capacidad de utilizar los recursos informáticos de manera eficiente para cumplir con los requisitos del sistema. Los temas clave incluyen la selección de los tipos y tamaños de recursos correctos en función de los requisitos de la carga de trabajo, la supervisión del rendimiento y la toma de decisiones informadas para mantener la eficiencia a medida que evolucionan las necesidades del negocio.
5- Optimización de costos: El pilar de optimización de costos incluye la capacidad de evitar o eliminar costos innecesarios o recursos subóptimos.

Información adicional:
Crear un sistema de software es muy parecido a construir un edificio. Si la base no es sólida, los problemas estructurales pueden socavar la integridad y la función del edificio. Al diseñar soluciones tecnológicas en Amazon Web Services (AWS), si descuida los cinco pilares de excelencia operativa, seguridad, confiabilidad, eficiencia de rendimiento y optimización de costos, puede ser un desafío crear un sistema que cumpla con sus expectativas y requisitos. La incorporación de estos pilares en su arquitectura ayuda a producir sistemas estables y eficientes. Esto le permite centrarse en los otros aspectos del diseño, como los requisitos funcionales. AWS Well-Architected Framework ayuda a los arquitectos de la nube a crear la infraestructura más segura, de alto rendimiento, resistente y eficiente posible para sus aplicaciones.

***Las otras opciones son incorrectas:***

***"La capacidad de un sistema para recuperarse con gracia de un fallo" es incorrecta.*** Esta afirmación está mucho más relacionada con el pilar de Fiabilidad.

***"El uso eficiente de los recursos informáticos para cumplir con los requisitos" es incorrecto.*** Esta declaración está mucho más relacionada con el pilar de Eficiencia del Desempeño.

***"La capacidad de administrar las operaciones del centro de datos de manera más eficiente" es incorrecta.*** La administración de las operaciones del centro de datos no está relacionada con ningún pilar. Es algo de lo que AWS es responsable NO del cliente.

**Equilibrio de carga:** el número de horas que se ejecuta Elastic Load Balancer y la cantidad de datos que procesa contribuyen al costo mensual de EC2.

**Tipo de instancia:** Amazon EC2 proporciona una amplia selección de tipos de instancias optimizadas para adaptarse a diferentes casos de uso. Los tipos de instancia comprenden diferentes combinaciones de CPU, memoria, almacenamiento y capacidad de red.

***Las otras opciones son incorrectas:***

***"La zona de disponibilidad donde se aprovisiona la instancia" es incorrecta.*** Los precios de las instancias Amazon EC2 pueden variar en función de la región en la que se aprovisionen las instancias. Las instancias Amazon EC2 aprovisionadas en diferentes zonas de disponibilidad dentro de la misma región tienen el mismo precio.

***"Número de IP privadas" es incorrecto.*** No hay ningún cargo por IP privadas.
Información adicional:
El número de IP elásticas asignadas es el factor que puede afectar a los cargos de Amazon EC2. Para garantizar un uso eficiente de las direcciones IP elásticas, AWS impone un pequeño cargo por hora si una dirección IP elástica **no está asociada** a una instancia en ejecución o si está **asociada** a una instancia detenida. Mientras se ejecuta la instancia, no se le cobra por una dirección IP elástica asociada a la instancia, pero las IP elásticas adicionales no son gratuitas.

***"Número de cubos" es incorrecto.*** Un bucket es un recurso de Amazon S3, no un recurso de Amazon EC2.
Información adicional:
Para cargar sus datos (fotos, vídeos, documentos, etc.) en Amazon S3, primero debe crear un bucket de S3 (que es como una carpeta de archivos) en una de las regiones de AWS. A continuación, puede cargar cualquier número de objetos en el bucket. Al cliente se le cobra en función del tamaño total de los objetos (en GB) almacenados en su bucket de S3, no para el bucket en sí.

Pregunta 59: **Omitido¿Qué proporciona Amazon ElastiCache?**
• **Un sistema de nombres de dominio en la nube**

• **Un almacén de datos en memoria compatible con Ehcache**
• **Una tienda de software en línea que permite a los clientes lanzar software preconfigurado con solo unos pocos clics
Explicación**
**ElastiCache** es un servicio web que facilita la configuración, administración y escalabilidad de un almacén de datos distribuido en memoria o un entorno de caché en la nube. Proporciona una solución de almacenamiento en caché de alto rendimiento, escalable y rentable, al tiempo que elimina la complejidad asociada con la implementación y administración de un entorno de caché distribuida. El almacenamiento en caché en memoria proporcionado por Amazon ElastiCache se puede utilizar para mejorar significativamente la latencia y el rendimiento de muchas aplicaciones de lectura pesada (como redes sociales, juegos, uso compartido de medios y portales de preguntas y respuestas) o cargas de trabajo de computación intensiva (como un motor de recomendaciones).
El almacenamiento en caché en memoria mejora el rendimiento de las aplicaciones al almacenar datos críticos en la memoria para un acceso de baja latencia. La información almacenada en caché puede incluir los resultados de consultas comunes de bases de datos o los resultados de cálculos computacionalmente intensivos.

Información adicional:
El propósito principal de un almacén de datos en memoria es proporcionar acceso ultrarrápido (latencia de submillisegundos) y económico a copias de datos. Consultar una base de datos siempre es más lento y costoso que localizar una copia de esos datos en una memoria caché. Algunas consultas de base de datos son especialmente costosas de realizar. Un ejemplo son las consultas que implican uniones entre varias tablas o consultas con cálculos intensivos. Al almacenar en caché (almacenar) dichos resultados de consulta, solo paga el precio de la consulta una vez. A continuación, puede recuperar rápidamente los datos varias veces sin tener que volver a ejecutar la consulta.

***"Un sistema de nombres de dominio en la nube" es incorrecto.*** Route53 es el servicio que proporciona DNS en la nube.

***"Un almacén de datos en memoria compatible con Ehcache" es incorrecto.*** ElastiCache solo admite dos motores de caché: Redis y Memcached.

Explicación**
Los gastos de capital (CapEx) son los principales gastos a largo plazo de una empresa, mientras que los gastos operativos (OpEx) son los gastos diarios de una empresa. Los ejemplos de CapEx incluyen activos físicos como edificios, equipos y maquinaria. Además, puede reducir los costos de gastos operativos (OpEx) relacionados con la administración y el mantenimiento de los datos. Pregunta 65: **Omitido¿Cuáles son las características de Amazon RDS que se pueden utilizar para mejorar la disponibilidad de su base de datos? (Elija DOS)**
• [ ] **Automatic patching (Parches automáticos)**
• [ ] **Multi-AZ Deployment (Implementación multi-AZ)(Correcto)**
• [ ] **Edge Locations (Ubicaciones de borde)**
• [ ] **Read Replicas(Leer replicas)(Correcto)**
• [ ] **(AWS Regions) Regiones de AWS
Explicación**
En una implementación Multi-AZ, Amazon RDS aprovisiona y mantiene automáticamente una réplica en espera sincrónica en una zona de disponibilidad diferente. La instancia de base de datos principal se replica sincrónicamente en todas las zonas de disponibilidad en una réplica en espera para proporcionar redundancia de datos, eliminar los bloqueos de E/S y minimizar los picos de latencia durante las copias de seguridad del sistema. La ejecución de una instancia de base de datos con alta disponibilidad puede mejorar la disponibilidad durante el mantenimiento planificado del sistema y ayudar a proteger las bases de datos contra errores de instancias de base de datos e interrupciones de la zona de disponibilidad.
Las réplicas de lectura de Amazon RDS proporcionan un rendimiento y una durabilidad mejorados para las instancias de base de datos (DB). Esta característica facilita la escalabilidad horizontal elástica más allá de las restricciones de capacidad de una única instancia de base de datos para cargas de trabajo de bases de datos de lectura intensiva. Puede crear una o más réplicas de una instancia de base de datos de origen determinada y servir tráfico de lectura de aplicaciones de gran volumen a partir de varias copias de los datos, lo que aumenta el rendimiento de lectura agregada.
Las réplicas de lectura proporcionan un mecanismo de disponibilidad complementario a las implementaciones multi-AZ de Amazon RDS. Puede promover una réplica de lectura si se produce un error en la instancia de base de datos de origen. También puede replicar instancias de base de datos en todas las regiones de AWS como parte de su estrategia de recuperación ante desastres. Esta funcionalidad complementa la replicación síncrona, la detección automática de fallos y la conmutación por error que se proporcionan con las implementaciones Multi-AZ.

***Las otras opciones son incorrectas:***

***"Ubicaciones de borde" es incorrecto.*** Las ubicaciones de borde no son una característica de Amazon RDS. El servicio CloudFront utiliza las ubicaciones de borde para distribuir contenido a nivel mundial.

***"Parches automáticos" es incorrecto.*** El propósito de la aplicación de parches es resolver problemas de funcionalidad, mejorar la seguridad o agregar nuevas características.

***"Regiones de AWS" es incorrecto.*** Las regiones de AWS no son una característica de Amazon RDS. Las regiones de AWS son áreas geográficas separadas en todo el mundo que AWS utiliza para proporcionar sus servicios en la nube, incluidas las regiones de América del Norte, América del Sur, Europa, Asia Pacífico y Oriente Medio. La elección de una región de AWS específica depende de su proximidad a los usuarios finales, la soberanía de los datos y los costos.

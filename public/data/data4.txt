Pregunta 1: **Omitido¿Qué servicio de AWS ayuda a las empresas a extender su almacenamiento local a AWS de una manera rentable?**
• **Amazon Aurora**
• **Amazon EFS**
• **AWS Data Pipeline**
• **AWS Storage Gateway(Correcto)
Explicación**
Las empresas pueden extender su almacenamiento local a la nube de AWS para la retención y el archivado de copias de seguridad a largo plazo, optimizando los costos y aumentando la resiliencia y la disponibilidad. AWS Storage Gateway es un servicio de almacenamiento híbrido que permite a las aplicaciones locales utilizar sin problemas el almacenamiento en la nube de AWS. Las empresas pueden usar el servicio para copias de seguridad y archivado, recuperación ante desastres, procesamiento de datos en la nube, organización en niveles de almacenamiento y migración. La gateway de almacenamiento se conecta a los servicios de almacenamiento de AWS, como Amazon S3, Amazon S3 Glacier, Amazon S3 Glacier Deep Archive, Amazon EBS y AWS Backup, que proporcionan almacenamiento para archivos, volúmenes, instantáneas y cintas virtuales en AWS.

***Las otras opciones son incorrectas:***

***Amazon Aurora es incorrecto.*** Amazon Aurora es un servicio de base de datos relacional compatible con MySQL y PostgreSQL.

***Amazon EFS es incorrecto.*** Amazon Elastic File System (Amazon EFS) proporciona un sistema de archivos NFS elástico totalmente administrado para su uso con los servicios en la nube de AWS y los recursos locales. Está diseñado para escalar bajo demanda a petabytes sin interrumpir las aplicaciones, creciendo y reduciéndose automáticamente a medida que agrega y elimina archivos, eliminando la necesidad de aprovisionar y administrar la capacidad para adaptarse al crecimiento. Aunque EFS se puede utilizar en entornos híbridos, no es tan rentable como Storage Gateway.

***AWS Data Pipeline es incorrecto.*** AWS Data Pipeline es un servicio web que ayuda a los clientes a procesar y mover datos de forma fiable entre diferentes servicios informáticos y de almacenamiento de AWS, así como orígenes de datos locales. AWS Data Pipeline no es un servicio de almacenamiento.

**Referencias:**
https://aws.amazon.com/storagegateway/

Pregunta 4: **Omitido¿Cuál de los siguientes NO es un factor a la hora de estimar los costes de Amazon EC2? (Elija DOS)**
• [ ] **La cantidad de tiempo que se ejecutarán las instancias**
• [ ] **Direcciones IP elásticas asignadas**
• [ ] **Número de grupos de seguridad(Correcto)**
• [ ] **Número de zonas hospedadas(Correcto)**
• [ ] **Número de instancias
Explicación**
No hay costos asociados para "Grupos de seguridad de EC2" o "Zonas alojadas" y, por lo tanto, son respuestas correctas.

Los grupos de seguridad de EC2 son de uso gratuito.

Las zonas alojadas no son gratuitas, pero no están relacionadas con los costos de Amazon EC2. Las zonas alojadas son uno de los factores de los costos de Amazon Route 53.

***Las otras opciones representan factores que debe tener en cuenta al estimar el costo de Amazon EC2 y, por lo tanto, son incorrectas.***

Cuando comience a estimar el costo del uso de Amazon EC2, tenga en cuenta lo siguiente:
**1- Horas de reloj de la hora del servidor:** La cantidad de tiempo que las instancias se ejecutarán tiene una relación directa con el precio general, ya que las instancias EC2 se cobran por hora o por segunda, dependiendo de qué AMI se utilice.
**2- Tipo de instancia:** Amazon EC2 proporciona una amplia selección de tipos de instancias optimizadas para adaptarse a diferentes casos de uso. Los tipos de instancia comprenden diferentes combinaciones de CPU, memoria, almacenamiento y capacidad de red.
**3- Modelo de precios:** On-Demand, Reservado, Spot, Planes de Ahorro y Dedicado
**4- Número de instancias:** puede aprovisionar varias instancias de sus recursos de Amazon EC2 y Amazon EBS para gestionar las cargas máximas.
**5- Equilibrio de carga:** El número de horas que se ejecuta el Elastic Load Balancer y la cantidad de datos que procesa contribuyen al costo mensual de EC2.
**6- Direcciones IP** elásticas: para garantizar un uso eficiente de las direcciones IP elásticas, AWS impone un pequeño cargo por hora si una dirección IP elástica no está **asociada** a una instancia en ejecución o si está **asociada** a una instancia detenida. Mientras se ejecuta la instancia, no se le cobra por una dirección IP elástica asociada a la instancia, pero las IP elásticas adicionales no son gratuitas.
**7- Sistemas operativos y paquetes de software:** Los precios de los sistemas operativos están incluidos en los precios de instancia, a menos que elija traer sus propias licencias.

**Referencias:**
https://docs.aws.amazon.com/whitepapers/latest/how-aws-pricing-works/how-aws-pricing-works.pdf

Según las prácticas recomendadas de AWS, la proximidad a sus usuarios finales, la conformidad normativa, las restricciones de residencia de datos y el costo son factores que debe tener en cuenta al elegir la región de AWS más adecuada.

***Las otras opciones son incorrectas:***

Pregunta 7: **Omitido¿Cuál de las siguientes opciones es una práctica recomendada para que su aplicación en AWS sea de alta disponibilidad?**
• **Implementar el código de la aplicación en al menos dos servidores de la misma zona de disponibilidad**
• **Reescribir el código de la aplicación para gestionar todas las solicitudes entrantes**
• **Uso de Elastic Load Balancing (ELB) en varias regiones de AWS**
• **Implementar la aplicación en al menos dos zonas de disponibilidad(Correcto)
Explicación**
Cada región de AWS contiene varias ubicaciones o zonas de disponibilidad distintas. Cada zona de disponibilidad está diseñada para ser independiente de los errores en otras zonas de disponibilidad. La implementación de la aplicación en varias zonas de disponibilidad aumentará la disponibilidad de la aplicación. Si una zona de disponibilidad encuentra un problema, las otras zonas de disponibilidad aún pueden servir a la aplicación.

***Las otras opciones son incorrectas:***

***"Usar Elastic Load Balancing (ELB) en varias regiones de AWS" es incorrecto.*** Elastic Load Balancing (ELB) es un servicio regional, no un servicio global. Elastic Load Balancing solo se puede utilizar para distribuir tráfico entre varias zonas de disponibilidad dentro de la misma región de AWS.

***"Implementar el código de la aplicación en al menos dos servidores de la misma zona de disponibilidad" es incorrecto.*** El uso de más servidores de AWS en la misma zona de disponibilidad ayudaría con el rendimiento siempre que la zona de disponibilidad no tuviera problemas, pero la implementación en una sola zona de disponibilidad constituye un único punto de error y, por lo tanto, no es una práctica recomendada.

***"Reescribir el código de la aplicación para manejar todas las solicitudes entrantes" es incorrecto.*** No existe relación entre el código de la aplicación y la "alta disponibilidad". Incluso el código perfectamente escrito que nunca se bloquea dejará de estar disponible si falla la infraestructura en la que se ejecuta.

**Referencias:**
https://aws.amazon.com/about-aws/global-infrastructure/regions_az/

Pregunta 13: **Omitido¿Cuáles son las ventajas de usar Auto Scaling Groups para instancias EC2?**
• **Auto Scaling Groups escala instancias EC2 en varias regiones para reducir la latencia de los usuarios globales**
• **Auto Scaling Groups distribuye el tráfico de aplicaciones entre varias zonas de disponibilidad para mejorar el rendimiento**
• **Auto Scaling Groups escala instancias EC2 en varias zonas de disponibilidad para aumentar la disponibilidad de las aplicaciones y la tolerancia a errores(Correcto)**
• **Auto Scaling Groups almacena en caché las respuestas más recientes en ubicaciones perimetrales globales para reducir la latencia y mejorar el rendimiento
Explicación**
**Amazon EC2 Auto Scaling** ofrece las siguientes ventajas:
1- **Mejor tolerancia a fallos**. Amazon EC2 Auto Scaling puede detectar cuándo una instancia no está en buen estado, terminarla y lanzar una instancia para reemplazarla. Además, Amazon EC2 Auto Scaling le permite aprovechar la seguridad y confiabilidad de la redundancia geográfica al abarcar grupos de Auto Scaling en varias zonas de disponibilidad dentro de una región. Cuando una zona de disponibilidad deja de estar en mal estado o no está disponible, Auto Scaling lanza nuevas instancias en una zona de disponibilidad no afectada. Cuando la zona de disponibilidad en mal estado vuelve a un estado correcto, Auto Scaling redistribuye automáticamente las instancias de la aplicación de manera uniforme en todas las zonas de disponibilidad designadas.
2- **Mejor disponibilidad**. Amazon EC2 Auto Scaling ayuda a garantizar que su aplicación siempre tenga la cantidad correcta de capacidad para gestionar la demanda de tráfico actual.
3- **Mejor gestión de costes**. Amazon EC2 Auto Scaling puede aumentar y disminuir dinámicamente la capacidad según sea necesario. Debido a que paga por las instancias EC2 que usa, ahorra dinero lanzando instancias cuando son necesarias y terminándolas cuando no lo son.

***Las otras opciones son incorrectas:***

***"Auto Scaling Groups distribuye el tráfico de aplicaciones entre varias zonas de disponibilidad para mejorar el rendimiento" es incorrecto.*** AWS ELB es el servicio que se utiliza para distribuir el tráfico. Los grupos de Auto Scaling no distribuyen el tráfico

***"Auto Scaling Groups escala instancias EC2 en varias regiones para reducir la latencia de los usuarios globales" es incorrecto.*** Un grupo de Auto Scaling puede contener instancias EC2 en una o más zonas de disponibilidad dentro de la misma región. Sin embargo, los grupos de Auto Scaling no pueden abarcar varias regiones.

***"Auto Scaling Groups almacena en caché las respuestas más recientes en ubicaciones de borde global para reducir la latencia y mejorar el rendimiento" es incorrecto.*** Amazon CloudFront es el servicio que se utiliza para almacenar en caché las respuestas más recientes en las ubicaciones de borde global para proporcionar un rendimiento más rápido a los usuarios globales. Los grupos de Auto Scaling no realizan ningún almacenamiento en caché.

Los roles de IAM le permiten delegar el acceso (durante un tiempo limitado) a usuarios, aplicaciones o servicios que normalmente no tienen acceso a sus recursos de AWS. Por ejemplo, es posible que desee conceder a los usuarios de su cuenta de AWS acceso a recursos que normalmente no tienen o conceder a los usuarios de una cuenta de AWS acceso a los recursos de otra cuenta. O bien, es posible que desee permitir que una aplicación móvil utilice recursos de AWS. A veces desea dar acceso a AWS a los usuarios que ya tienen identidades definidas fuera de AWS, como en su directorio corporativo. O bien, es posible que desee conceder acceso a su cuenta a terceros para que puedan realizar una auditoría de sus recursos. Para estos escenarios, puede delegar el acceso a los recursos de AWS mediante un rol de IAM.

**Usuarios de IAM:**
Un usuario de IAM es una entidad que se crea en AWS para representar a la persona o el servicio que lo utiliza para interactuar directamente con AWS. Un uso principal para los usuarios de IAM es conceder a las personas acceso a la consola de administración de AWS para tareas interactivas y/o realizar solicitudes programáticas a los servicios de AWS mediante la API o la CLI. Un usuario en AWS consta de un nombre, una contraseña para iniciar sesión en la consola de administración de AWS y hasta dos claves de acceso que se pueden utilizar con la API o la CLI. Cuando se crea un usuario de IAM, se le conceden permisos convirtiéndolo en miembro de un grupo que tiene políticas de permisos adecuadas asociadas (recomendado) o adjuntando directamente políticas al usuario.

Información adicional:
Un rol de IAM es similar a un usuario de IAM, ya que es una identidad de AWS con políticas de permisos que determinan lo que la identidad puede y no puede hacer en AWS. Sin embargo, en lugar de estar asociado de forma única con una persona, un rol está destinado a ser asumido por cualquier persona (o cualquier servicio, aplicación, ... etc) quien lo necesita. Además, un rol no tiene credenciales estándar a largo plazo, como una contraseña o claves de acceso asociadas. En su lugar, cuando asume un rol, le proporciona credenciales de seguridad temporales para su sesión de rol. Los roles de IAM están destinados a ser asumidos por entidades autorizadas, como usuarios de IAM, aplicaciones o un servicio de AWS como EC2.

***Las otras opciones son incorrectas:***

***"AWS Organizations" es incorrecto.*** AWS Organizations es un servicio de administración de cuentas que le permite consolidar varias cuentas de AWS en una organización que cree y administre de forma centralizada.

***"Políticas de IAM" es incorrecto.*** Las políticas de IAM le permiten permitir o denegar el acceso a los servicios de AWS (como Amazon S3), recursos individuales de AWS (como un bucket de S3 específico) o acciones de API individuales (como s3:CreateBucket). Una política de IAM solo se puede aplicar a usuarios, grupos o roles de IAM y nunca puede restringir la identidad raíz de la cuenta de AWS (la cuenta raíz de AWS). Es importante tener en cuenta que, si bien las políticas de IAM son utilizadas por IAM Identities, la política en sí no es una forma de identidad de IAM.

Pregunta 34: **OmitidoUna organización tiene una aplicación heredada diseñada utilizando una arquitectura basada en monolíticos. ¿Qué servicio de AWS se puede utilizar para desacoplar los componentes de la aplicación?**
• **AWS Artifact**
• **Amazon SQS(Correcto)**
• **Virtual Private Gateway**
• **Amazon CloudFront
Explicación**
Una aplicación monolítica está diseñada para ser autónoma; los componentes de la aplicación están interconectados e interdependientes en lugar de estar poco acoplados, como es el caso de las aplicaciones de microservicios.
Con arquitecturas monolíticas, todos los procesos están **estrechamente acoplados** y se ejecutan como un solo servicio. Esto significa que si un proceso de la aplicación experimenta un aumento en la demanda, toda la arquitectura debe escalarse. Agregar o mejorar las características de una aplicación monolítica se vuelve más complejo a medida que crece la base de código. Esta complejidad limita la experimentación y dificulta la implementación de nuevas ideas. Las arquitecturas monolíticas agregan riesgo para la disponibilidad de las aplicaciones porque muchos procesos dependientes y estrechamente acoplados aumentan el impacto de una falla de un solo proceso.
Con una arquitectura de microservicios, una aplicación se crea como componentes **de acoplamiento flexible**que ejecutan cada proceso de aplicación como un servicio. Estos servicios se comunican a través de una interfaz bien definida utilizando API ligeras. Los servicios se crean para las capacidades empresariales y cada servicio realiza una sola función. Debido a que se ejecutan de forma independiente, cada servicio se puede actualizar, implementar y escalar para satisfacer la demanda de funciones específicas de una aplicación. Las arquitecturas de microservicios hacen que las aplicaciones sean más fáciles de escalar y más rápidas de desarrollar, lo que permite la innovación y acelera el tiempo de comercialización de nuevas características.

**Los servicios de AWS que pueden ayudarle a crear aplicaciones de acoplamiento flexible incluyen:**

**1- Amazon Simple Queue Service (SQS):** Amazon SQS es un servicio de cola de mensajes totalmente administrado que le permite desacoplar y escalar microservicios, sistemas distribuidos y aplicaciones sin servidor. Amazon SQS ofrece una cola alojada fiable y altamente escalable para almacenar mensajes a medida que viajan entre aplicaciones o microservicios. Pregunta 44: **Omitido¿Cuáles de las siguientes son las ventajas de usar AWS como proveedor de computación en la nube? (Elija DOS)**
• [ ] **Elimina la necesidad de adivinar las necesidades de capacidad de la infraestructura(Correcto)**
• [ ] **Elimina la necesidad de supervisar servidores y aplicaciones**
• [ ] **Gestiona todas las tareas de cumplimiento y auditoría**
• [ ] **Proporciona hardware personalizado para cumplir con cualquier especificación**
• [ ] **Permite a los clientes intercambiar sus gastos de capital por gastos operativos(Correcto)
Explicación**
**Las ventajas de la computación en la nube incluyen: (IMPORTANTE)**
**1- Intercambie capital por gastos variables:** En lugar de tener que invertir mucho en centros de datos y servidores antes de saber cómo los va a usar, solo puede pagar cuando consume recursos informáticos y solo pagar por cuánto consume. Al utilizar AWS, los costos de infraestructura se convierten en un modelo de pago por uso, donde se cobra a los clientes por los recursos que consumen y esos costos se incurren como costos operativos en lugar de como gastos de capital.
**2- Benefíciese de las economías de escala masivas:** Al usar la computación en la nube, puede lograr un costo variable más bajo que el que puede obtener por su cuenta. Debido a que el uso de cientos de miles de clientes se agrega en la nube, los proveedores como Amazon Web Services pueden lograr mayores economías de escala, lo que se traduce en precios de pago más bajos.
**3- Deje de adivinar la capacidad:** Elimine las conjeturas sobre las necesidades de capacidad de su infraestructura. Cuando toma una decisión de capacidad antes de implementar una aplicación, a menudo termina sentado en costosos recursos inactivos o lidiando con una capacidad limitada. Con la computación en la nube, estos problemas desaparecen. Puede acceder tanto o tan poco como necesite, y escalar hacia arriba y hacia abajo según sea necesario con solo unos minutos de anticipación.
**4- Aumente la velocidad y la agilidad:** en un entorno de computación en la nube, los nuevos recursos de TI están a solo un clic de distancia, lo que significa que reduce el tiempo que lleva poner esos recursos a disposición de sus desarrolladores de semanas a solo minutos. Esto resulta en un aumento dramático en la agilidad para la organización, ya que el costo y el tiempo que lleva experimentar y desarrollar es significativamente menor.

El escalado horizontal significa agregar varias instancias más pequeñas cuando aumentan las cargas de trabajo, en lugar de agregar capacidad adicional de CPU, memoria o disco a una sola instancia. En la sintaxis de esta pregunta, ejecutar varias instancias EC2 en paralelo logra escalabilidad horizontal y es la respuesta correcta.
AWS recomienda que los clientes escalen los recursos horizontalmente para aumentar la disponibilidad agregada del sistema. Reemplazar un recurso grande con múltiples recursos pequeños en paralelo reducirá el impacto de una sola falla en el sistema general. Por ejemplo, si un cliente desea convertir un gran número de archivos binarios en archivos de texto o transcodificar un gran número de archivos de vídeo a otro formato, se recomienda que utilice varias instancias EC2 en paralelo en lugar de utilizar una instancia grande.

***Las otras opciones son incorrectas:***

***"Escalar verticalmente instancias EC2" es incorrecto.*** Se recomienda el escalado horizontal sobre el escalado vertical.

***"Escalar verticalmente instancias de RDS" y "Ejecutar instancias de RDS en paralelo" son incorrectos.*** Las instancias de RDS se usan para almacenar y ejecutar bases de datos y no se usarían para el procesamiento de archivos.

Pregunta 57: **OmitidoUna de las principales ventajas de usar AWS es el ahorro de costos. ¿Qué proporciona AWS para reducir el costo de ejecución de instancias Amazon EC2?**

Pregunta 58: **Omitido¿Qué entidad de IAM se puede utilizar mejor para conceder acceso temporal a sus recursos de AWS?**
• **IAM Roles(Correcto)**
• **Usuarios de IAM**
• **Grupos de IAM**
• **Par de claves
Explicación**
Un rol de IAM es una entidad de IAM que define un conjunto de permisos para realizar solicitudes de servicio de AWS. Los **roles de IAM no están asociados a un usuario o grupo específico**. En su lugar, las entidades de confianza asumen roles, como usuarios de IAM, aplicaciones o servicios de AWS como EC2. Un rol de IAM es similar a un usuario de IAM, ya que es una identidad de AWS con políticas de permisos que determinan lo que la identidad puede y no puede hacer en AWS. Sin embargo, en lugar de estar asociado de manera única con una persona, un rol está destinado a ser asumido por cualquiera que lo necesite. Además, un rol no tiene credenciales estándar a largo plazo, como una contraseña o claves de acceso asociadas. En su lugar, cuando asume un rol, le proporciona credenciales de seguridad temporales para su sesión de rol.
Puede utilizar roles para delegar el acceso a usuarios, aplicaciones o servicios que normalmente no tienen acceso a sus recursos de AWS. Por ejemplo, es posible que desee conceder a los usuarios de su cuenta de AWS acceso a recursos que normalmente no tienen o conceder a los usuarios de una cuenta de AWS acceso a los recursos de otra cuenta. O bien, es posible que desee permitir que una aplicación móvil utilice recursos de AWS, pero no desee incrustar claves de AWS en la aplicación. A veces desea dar acceso a AWS a los usuarios que ya tienen identidades definidas fuera de AWS, como en su directorio corporativo. O bien, es posible que desee conceder acceso a su cuenta a terceros para que puedan realizar una auditoría de sus recursos. Para estos escenarios, puede delegar el acceso temporal a los recursos de AWS mediante un rol de IAM.

***Las otras opciones son incorrectas.***

***"Usuarios de IAM" es incorrecto.*** Un usuario de IAM tiene credenciales permanentes a largo plazo, no credenciales temporales.

***"Grupos de IAM" es incorrecto.*** Un grupo de IAM es una forma de administrar lógicamente conjuntos de usuarios de IAM que requieren permisos idénticos.

***"Par de claves" es incorrecto.*** El par de claves de Amazon EC2 le permite acceder de forma segura a sus instancias mediante una clave privada en lugar de una contraseña. Puede crear y descargar este par de claves al lanzar una nueva instancia EC2.

Información adicional:
El par de claves es diferente de las claves de acceso de AWS. Las claves de acceso son credenciales de seguridad (como un nombre de usuario y una contraseña) que permiten a los usuarios interactuar con los servicios de AWS mediante programación.

En su lugar, debe abrir una investigación, comprobar los registros de AWS CloudTrail y eliminar todos los usuarios de IAM potencialmente comprometidos.
